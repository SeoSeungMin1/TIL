## 배운 내용

- Synchronized
    - 데이터 충돌 과정에서 가장 기본적으로 제공
    - 여러 스레드가 동시에 공유 자원에 접근하지 못하도록 일시적으로 해당 자원에 대한 접근을 하나로 제한하는 동기화 키워드
    - 임계 영역 : 여러 스레드가 동시에 실행해서는 안되는 프로그램 코드의 일부분(구간)
    - 스레드가 임계 영역에 진입하면 자동으로 lock 획득, 해당 코드 실행 끝나면 lock 반납하여 다른 스레드가 대기 후 진입할 수 있게 함.
    - Lock : 동시에 여러 스레드가 하나의 공유 자원에 접근하지 못하게 막는 장치.
    - 사용 이유 : 멀티스레드 환경에서 공유 자원의 충돌 방지, 실행 흐름의 안정성과 데이터 일관성 보장.
    - 실무에서 사용 이유 : 공유 자원 보호, 무증상 오류 예방
    - 무증상 오류 : 오류가 발생했지만 예외나 경고 없이 정상처럼 보이는 오류
    - JVM이 자동으로 lock획득 → 코드 실행 → lock 해제함.
    - 기본 형식 : 메서드나 코드 블록에 적용
        
        → 메서드 방식 : 메서드 전체 lock, 블록 방식 : 특정 코드만 lock
        
        → lock 기준 객체 : this, 특정 객체(Object), 클래스 객체
        
    - lock 기준 객체가 다르면 동시에 lock 가능. → 불필요한 대기 줄이고 병렬성 높이기 위해서 작업마다 lock 객체를 따로 만듬.
    - 흐름 : CPU → 스레드 실행 → run() / call() 코드 실행 → Synchronized 만나면 lock객체 확인 → 획득하면 실행하고 lock 해제, 획득 못할 시 다른 스레드가 이미 lock 사용중이어서 BLOCKED 상태로 대기.
- Lock
    - Synchronized가 자동문이면 Lock은 직접 열쇠를 쥐고 원하는 타이밍에 문을 잠그고 열 수 있는 수동 자물쇠.
    - 멀티스레드 환경에서의 안전한 자원 접근을 보장하기 위해 사용되는 동기화 제어 인터페이스
    - 락의 적용 시점과 해제 시점, 락의 대상 객체까지 직접 제어.
    - 비차단 락(non-blocking lock) : 스레드가 락을 즉시 획득하지 못하면 기다리지 않고 바로 다음 로직으로 넘어가는 방식
    - lock 획득 성공 : 락 획득 → 임계 영역(공유 데이터 작업) 실행 → 작업 종료 → unlock()으로 lock 해제
    - lock 획득 실패 : Synchronized처럼 대기하지 않고 다른 로직 수행
    - 개발자가 lock 획득과 해제 직접 관리
    - 사용 이유 : 멀티스레드 환경에서 자원 접근을 정밀하게 제어하고, 안정성과 예측 가능성을 확보할 수 있음.
    - 사용 방법 : lock.lock()으로 락 획득, 작업 수행하고 반드시 finally 블록에서 lock.unlock()으로 해제.
    - finally 블록에서 해제하는 이유 : catch는 예외 발생했을 때만 실행하는데 예외 발생 여부와 상관없이 무조건 lock 해제해야 하기 때문.
    - 비차단 락 - tryLock() : 지정한 시간 동안만 락 획득 시도, 실패하면 기다리지 않고 다른 로직 이동
        
        → 락을 획득하지 못했을 경우 우회하거나 재시도 로직 명확하게 처리.
        
    - ReentrantLock(true) : 락을 먼저 요청한 스레드가 먼저 락을 획득.
        
        → 특정 스레드가 계속해서 락 획득 못하고 무한 대기하는 상태인 기아 상태 방지.
        
    - Condition : Lock과 함께 사용하는 객체, 공유 데이터의 특정 조건이 만족될 때까지 스레드를 대기시키고 다시 깨우는 동기화 기능
        
        → 생성 방법
        
        ```java
        Lock lock = new ReentrantLock();
        Condition condition = lock.newCondition();
        ```
        
    - ThreadLocal : 스레드마다 고유하게 독립된 변수를 저장할 수 있도록 해주는 클래스.
        
        → 스레드마다 독립된 값을 갖기 때문에 동기화 없이도 thread-safe하게 데이터 저장
        
- Volatile
    - 변수의 값을 모든 스레드가 항상 메인 메모리에서 직접 읽고 쓰도록 보장하는 키워드
    - Volatile가 붙은 변수는 항상 메인 메모리에서 직접 읽고 쓰도록 강제하여 다른 스레드들도 변경된 최신 값을 즉시 읽음.
    - 사용이유 : 스레드 간 변수 변경 내용을 즉시 공유하고, 값의 최신 상태를 예측 가능하게 유지하기 위해
        
        → Synchronized나 Lock 사용해도 되는데 Volatile 사용 이유
        
        → lock 없이 가시성만 보장하기 때문에 성능 부담이 적고 코드 간결함.
        
    - 사용하지 말아야하는 경우
        
        → volatile은 최신값(가시성)은 보장하지만, 연산 전체를 하나로 보호하지 못하므로 원자성은 보장하지 않는다.
        
        → 여러 스레드가 동시에 값을 읽고 수정하는 count++ 같은 복합 연산에 적합하지 않음. (1. 값을 읽고 2. 1을 더하고 3. 다시 저장하는 3단계 작업)
        
        → 한 스레드가 값을 변경하고 다른 스레드가 값을 읽기만 하는 플래그 변수(상태만 표시하는 변수)에 적합
        
    - 가시성 : 한 스레드가 변경한 값을 다른 스레드가 즉시 최신  상태로 볼 수 있는 성질
    - 원자성 : 하나의 연산이 중간에 다른 스레드에 의해 끼어들지 않고, 처음부터 끝까지 하나의 작업처럼 실행되는 성질
- Atomic Type
    - 멀티스레드 환경에서 경쟁 조건 없이 값의 읽기, 쓰기, 수정이 안전하게 보호되도록 설계된 타입
    - 경쟁 조건 없이 안전한 이유
        
        → 내부적으로 CAS 연산을 사용하기 때문에.
        
        → CAS 동작
        
        1. 현재 메모리 값(V)
        2. 예상 값(E)
        3. 새로 바꿀 값(N)
        
        ```java
        if (V == E) {
            V ← N
        } else {
            실패 → 다시 시도
        }
        ```
        
    - 경쟁 조건 : 여러 스레드가 동시에 공유 데이터에 접근하면서, 실행 순서에 따라 결과가 달라지는 문제
    - 사용 이유 : 멀티스레드 환경에서 변수에 대한 안전한 연락을 락 없이 처리할 수  있기 때문
    - 선언 : private AtomicInteger 변수이름 = new AtomicInteger(0);
        
        →AtomicInteger가 내부적으로 CAS 연산 사용해  공유 데이터를 원자적으로 안전하게 수정
        
- 정리

| 항목 | volatile | Atomic Type | Synchronized/Lock |
| --- | --- | --- | --- |
| 목적 | 가시성 보장 | 원자성 + 가시성 보장 | 원자성 + 가시성 보장 |
| 동작 방식 | 최신 값만 즉시 반영 | CAS 기반 연산 | lock 기반 동기화 |
| 사용 대상 | 단순 상태값, 플래그 변수 | 단일 변수 연산 | 여러 변수, 복잡한 로직 |
| 표현 방식 | volatile boolean stop | AtomicInteger | synchronized, ReentrantLock |
| 적합한 상황 | 읽기/쓰기만 있는 경우 | 단순 연산 동시 처리 | 복합 객체 상태 처리 |

## 헷갈리는 점

- 동시성, 가시성, 원자성에 개념이 헷갈려서 반복해서 볼 필요가 있다고 느꼈다.
- volatile, Atomic Type, Synchronized, Lock을 상황에 따라 적합하게 사용해야 되는데 언제 어떤걸 사용해야될 지 헷갈린다.

## 느낀 점

멀티스레드 환경에서는 여러 스레드가 공유 데이터를 동시에 사용하기 때문에 동시성 문제와 가시성 문제가 발생할 수 있다는 점을 이해할 수 있었습니다. 또한 상황에 따라 volatile, Atomic Type, synchronized, Lock 등을 적절하게 사용하여 공유 변수를 안전하게 보호하는 것이 중요하다고 느꼈습니다.
