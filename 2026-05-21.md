## 배운 내용

- ReenntrantLock
    - 동일한 스레드가 여러 번 안전하게 진입할 수 있는 락
    - 같은 스레드가 이미 획득한 락을 다시 획득할 수 있도록 허용하는 락 구현체
    - 사용 이유
        
        → synchronized 키워드 보다 더 정밀한 락 제어가 필요한 경우
        
        → 동일한 스레드가 이미 획득한 락을 다시 요청할 경우 예외 없이 재진입 허용
        
        → 명시적이고 정교한 동기화 제어 가능.
        
    - 사용 방법
        
        → lock()으로 락을 획득하고 unlock()으로 해제
        
        → try-finally 구문을 활용해 예외 발생 시에도 락 해제를 보장.
        
        → tryLock() : 비차단 락 - 대기 없이 락을 시도 / 기다리지 않음
        
        → lockInterruptibly() : 인터럽트 대응 락 - 락 기다리다가 인터럽트 오면 중단 가능
        
    - Synchronized vs ReentrantLock
    
    | **항목** | synchronized | ReentrantLock |
    | --- | --- | --- |
    | 락 획득 방법 | 자동 (블록 진입 시 획득) | 명시적 (lock(), unlock() 사용) |
    | 락 해제 방법 | 자동 (블록 종료 시 해제) | 명시적 (unlock() 호출 필요) |
    | 락 획득 대기 중 인터럽트 처리 | 불가능 (인터럽트 무시) | 가능 (lockInterruptibly() 사용 가능) |
    | 락 획득 시도 및 실패 처리 | 불가능 | 가능 (tryLock()으로 대기 없이 시도 가능) |
    | 조건 변수 (Condition) 사용 | 불가능 (wait/notify만 사용 가능) | 가능 (newCondition()으로 세분화 가능) |
    | 공정성 (FIFO 보장) 설정 | 불가능 | 가능 (new ReentrantLock(true)) |
    | 재진입 가능 여부 | 가능 | 가능 |
- Reflection
    - 클래스, 메서드, 필드 등의 정보를 실행중(runtime)에 동적으로 조회하고 조작할 수 있게 해주는 기능
    - 자바는 정적 타입 언어이기 때문에, 어떤 클래스나 메서드를 호출할지는 컴파일 시점에 대부분 결정.
    - `java.lang.reflect` 패키지를 통해 제공.
    - 정적 타입 언어 : 변수의 타입이 컴파일 시점에 결정되며, 타입을 명시적으로 선언해야 된다.
        
        → 컴파일러가 코드를 분석하면서 타입 오류를 미리 확인할 수 있기 때문에 실행 전에 문제가 감지되어 안정성과 성능이 높다.
        
        → Java, C, C++ 등
        
    - 동적 타입 언어 : 변수의 타입이 실행 시점(runtime)에 결정되며, 타입을 사전에 선언하지 않아도 된다.
        
        → 빠른 개발과 높은 유연성을 제공하지만, 타입 오류가 프로그램 실행 중에 발생할 수 있어 런타임 오류의 위험이 크다.
        
        → Python, JavaScript 
        
    - 사용 이유
        
        → 정해지지 않은 구조나 외부 입력에도 유연하게 대응할 수 있는 확장성과 자동화를 위해
        
        → 접근 제한된 멤버에 강제로 접근
        
    - 단점
        
        → 성능이 엄청 느리다 - 런타임에 동적으로 메모리를 뒤져서 클래스 정보, 메서드, 필드를 찾아와야해서 일반적인 메서드 호출보다 훨씬 느리다.
        
        → 컴파일 시점에 에러를 잡을 수 없다. - 오타나 클래스가 없어도 컴파일은 성공하는데 런타임 오류가 발생할 수 있다. → 서비스 실행 중 터질 가능성 증가
        
        → 객체지향 캡슐화 파괴 - private이라고 하는 규칙이 있는데 무시하고 접근 할 수 있다는 건 좋지 않다.
        
    - 비즈니스 로직을 테스트 하기 위한 테스트 코드 작성할때만 제한적으로 쓸 수 있다.

## 느낀 점

멀티스레드 환경에서 상황에 맞게 Synchrnized와 ReentrantLock 같은 동기화 방식을 적절히 선택하는 것이 중요하다는 점을 느꼈다. 또한 Reflection은 성능 저하, 런타임 오류 가능성, 캡슐화 파괴 등의 단점이 있어서 테스트 코드 작성처럼 필요한 상황에서 제한적으로 사용해야 한다는 점을 배울 수 있었다.
