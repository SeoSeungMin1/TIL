## 배운 내용

- INTERNET
    - 하드웨어와 소프트웨어 인프라의 집합체
    - TCP/IP라는 표준 프로토콜 사용하여 다양한 컴퓨터 네트워크들이 서로 통신할 수 있게 함.
    
    TCP/IP - 인터넷에서 데이터를 전송하기 위해 사용되는 기본 통신 프로토콜
    
    - 브라우저에서 [google.com](http://google.com) 입력 시 동작 과정
        1. 사용자가 브라우저에 [google.com](http://google.com) 입력 → 컴퓨터는 해당 도메인의 IP주소를 알아내기 위해 DNS 서버에 요청 보냄 → google.com에 해당하는 IP 주소 반환
        2. IP주소를 알게 되면 요청이 내부 네트워크를 지나 게이트웨이 통과.
        3. 이후 요청 데이터는 패킷 단위로 나뉘어 인터넷으로 전송. 각 패킷에는 출발지 IP와 목적지 IP 포함, 여러 라우터를 거치면서 가장 효율적으로 경로를 따라 구글 서버로 이동
        4. 구글 서버는 요청을 받으면 홈페이지를 구성하는 HTML, CSS, JavaScript 파일 등을 응답 패킷 형태로 다시 사용자에게 전송
        5. 응답 패킷은 여러 네트워크 장비와 게이트웨이를 거쳐 사용자 컴퓨터 도착 → 브라우저는 이를 조립하여 최종적으로 구글 홈페이지 화면 사용자에게 보여줌
    - 인터넷 vs 웹
        
        → 인터넷 : 전 세계 컴퓨터를 연결한 거대 네트워크
        
        → 웹 : 인터넷 위에서 동작하는 서비스 중 하나, 브라우저를 통해 보는 서비스
        
    - Gateway : 서로 다른 네트워크 또는 시스템 간 연결점 또는 진입점 역할을 하는 물리적인 장치나 소프트웨어
        
        → 인트라넷이랑 인터넷 연결하는 역할만 하는게 아님
        
- 클라이언트 - 서버 구조
    
    
    | 왕과 장군 예시 단어 | 고유 명사 |  |
    | --- | --- | --- |
    | 장군 | client | 요청을 보내는 주체/ 웹 브라우저나 스마트폰 앱 |
    | 왕 | web server | 요청을 받아서 처리하고 응답하는 주체/ 예시에서 요청을 듣고 물자를 반환 |
    | 병사 | Client Side Language | 클라이언트의 요청을 만들고 전달하는 역할/ HTML, CSS, JS |
    | 성 | server | 왕이 있는 물리적 공간/ 웹 서버 프로그램이 실행되고 있는 물리적인 컴퓨터나 가상머신 |
    | 좌표 (성의 위치) | ip | 고유한 숫자 주소. 숫자로만 되있어서 사람이 외우기 어렵다. |
    | 도로명주소 | domain | 사람이 이해하기 쉬운 문자 주소. → DNS는 도메인과 IP를 연결하고 매칭시켜주는 역할. |
    | 성벽 | firewall | 외부 비정상적인 접근을 막는 보안 시스템. 해킹이나 악성 트래픽으로부터 서버를 보호 |
    | 성문 | port | 서버에서 특정 서비스로 들어가는 통로 |
    | 마패 | 인증 | 요청자의 신원을 확인하는 절차 |
    | 요청서 | http message | 클라이언트가 서버에게 보내는 구체적인 요청 내용 → 로그인 해달라, 상품 정보 보여달라 라는 메세지 등 |
    | 신하 | Server Side Language | 서버의 내부에서 실질적인 작업을 처리하는 역할/ Java, Python, Ts(nodejs) |
    | 창고 | Database | 데이터를 체계적으로 저장하고 관리하는 공간 |
    | 약속한 요청 규칙 | Protocol | 클라이언트랑 서버가 서로 통신하기 위한 약속, 규칙/ 웹은 HTTP라는 프로토콜을 사용해 정해진 행식으로 메시지 주고 받음. |
    - Client Side Language
        
        → HTML : 홈페이지 레이아웃을 만들어주는 컴퓨터 언어
        
        → CSS : 홈페이지 스타일을 입혀줄 떄 사용하는 컴퓨터 언어
        
        → JavaScript : 홈페이지에서 동적인 동작을 수행할 수 있도록 하는 프로그래밍 언어
        
    - Server Side Language
        
        → Java : 안정성과 보안성이 높아 대규모 웹 개발에서 주로 사용
        
        → PHP : 간단한 웹 페이지 개발할 때 사용
        
        → JavaScript : 실시간으로 정보를 주고 받는 웹 페이지 개발할 때 사용
        
        → Python : 웹 개발 뿐 아니라 다양한 분야에서 사용되는 언어
        
    - Port : 서버 안에서 어떤 프로그램과 통신할지 구분하는 번호
        
        → 80 : HTTP
        
        → 443 : HTTPS
        
        → 22 : SSH
        
        → 필요한 포트만 열어두고 나머지는 닫고 관리
        
    - Protocol : 통신 규약
        
        → 웹에서는 HTTP 기반으로 요청을 주고 받음
        
        → HTTP 특징 : stateless하다. → 서버는 클라이언트의 이전 요청을 기억하지 못한다.
        
        → 모든 요청은 독립적으로 취급
        
    - 인증 : 시스템에 등록된 사용자가 맞는지 확인하는 절차
    - 인가 : 인증된 사용자가 무엇을 할 수 있는지 권한 확인
        
        → 인증되지 않은 사용자 접근 막고, 인증된 사용자라도 인가되지 않은 작업을 막아야한다.
        
- 클라이언트 - 서버 응용
    - HTTPS 통신 과정 설명
        1. 도메인 이름 입력 - 사용자가 브라우저에 `https://google.com` 같은 주소 입력. → HTTPS는 암호화된 통신 방식
        2. DNS 조회 - DNS 서버에 [google.com](http://google.com/)의 IP주소를 요청
        3. Forward Proxy 통과(선택적) - 회사나 학교 환경에서는 보안이나 트래픽 관리를 위해
        4. TCP 3-Way Handshake - IP 주소를 알게 되면 클라이언트와 서버는 TCP 연결을 맺기 위해 3-way handshake를 수행
        5. TLS 핸드셰이크 시작 - TCP 연결이 완료되면 HTTPS 암호화를 위한 TLS 핸드셰이크를 시작
        6. 서버의 공개키 및 인증서 전달 - 서버는 자신의 SSL/TLS 인증서와 공개키를 브라우저에게 전달한다. 인증서에는 서버 정보와 공개키가 포함되어 있다.
        7. CA 인증서 검증 - 브라우저는 인증서를 발급한 CA(Certificate Authority)의 공개키를 이용해 인증서가 진짜인지 확인한다. 이를 통해 위조 서버 여부를 검증한다.
        8. 암호화된 세션 생성 - 인증서 검증이 완료되면 브라우저와 서버는 공유 비밀키(Session Key)를 생성한다. 이후 통신은 이 키를 이용해 암호화된다.
        9. 안전한 HTTPS 통신 - 암호화된 연결이 완료되면 브라우저와 서버는 안전하게 데이터를 주고받는다. 로그인 정보나 개인정보도 암호화되어 전송된다.
    - CDN : 사용자에게 가장 가까운 서버에서 이미지, 영상, 파일 등을 대신 전달해주는 시스템
    - 모놀리식 : 회원, 상품, 주문, 결제 같은 모든 기능을 하나의 큰 프로젝트와 하나의 서버에서 함께 운영하는 구조이다.
        
        → 코드가 한 곳에 모여 있어 개발과 테스트가 쉽다.
        
        → 서버 배포 과정이 단순
        
        → 빠르게 서비스 출시
        
        → 대부분의 서비스는 처음에 모놀리식 구조로 시작 → 서비스 규모가 커지고 사용자가 많아지면 한계 발생
        
    - MSA : 모놀리식의 문제를 해결하기위해 등장한 구조 → 기능별로 서비스를 잘게 분리
        
        → 각각 독립적인 애플리케이션으로 운영 → 클라이언트 요청은 API Gateway가 받아 적절한 서비스로 전달
        
        → 장점 
        
        - Fault Isolation(장애 격리) - 결제 서버가 다운되어도 상품, 회원 서버는 정상 작동 가능 → 상품 조회, 장바구니 기능 등은 계속 사용할 수 있어 전체 마비를 방지
        - 독립적인 확장 - 필요한 서비스만 확장할 수 있어 자원을 효율적으로 사용할 수 있다.
        - 기술 선택의 유연성 - 서비스마다 다른 기술 사용이 가능하다. 다만 실제 회사에서는 유지보수와 인력 문제 때문에 기술 스택을 너무 다양하게 사용하지는 않는 경우가 많다.
- 네트워크 모델 - 네트워크 프로토콜
    - 네트워크 상에서 데이터를 주고받는 규칙과 절차를 정의하는 체계
    - 사용 이유 : 데이터를 안전하고 신뢰성 있게 전송하기 위해
    - OSI 7계층
        1. 물리 계층 - 전기 신호 전달
            
            → 랜선, 광케이블, 허브
            
        2. 데이터 링크 계층 - 같은 네트워크 안에서 전달
            
            → MAC 주소, 스위치
            
        3. 네트워크 계층 - 목적지까지 경로 찾기
            
            → IP, 라우터
            
        4. 전송 계층 - 데이터를 안전하게 전달
            
            → TCP, UDP
            
        5. 세션 계층 - 연결 유지 관리
            
            → 로그인 유지
            
            → 통신 연결 생성, 종료
            
        6. 표현 계층 - 데이터 형식 변환
            
            → 암호화, 압축, 인코딩
            
        7. 응용 계층 - 사용자가 직접 사용하는 계층
            
            → 웹 브라우저, 카카오톡, 이메일
            
            → 프로토콜 : HTTP, HTTPS, FTP
            
    - TCP/IP 4계층
        1. 네트워크 인터페이스 계층 - 실제 네트워크 전송
            
            → MAC 주소, 랜카드, 스위치, 이더넷
            
            → OSI의 데이터링크, 물리 합친 계층
            
        2. 인터넷 계층 - 목적지 주소 찾기
            
            → IP, ICMP, ARP
            
            → OSI의 네트워크 계층
            
        3. 전송 계층 - 데이터 전달 방식
            
            → TCP, UDP
            
        4. 응용 계층 - 사용자가 사용하는 프로그램/서비스
            
            → 웹 브라우저, 카카오톡, 이메일
            
            → HTTP, HTTPS, FTP, DNS
            
            → OSI의 응용, 표현, 세션 합친 계층
            
    - TCP
        
        → 연결 지향형 프로토콜, 신뢰성 있는 전송
        
        → 3-way handshake 기반으로 통신 연결 맺고, 4-way handshake 갖고 연결 마친다. → 주고 받는 과정에서 신뢰성 높음
        
        → 데이터 완전성이 중요하면 TCP 기반의 프로토콜
        
    - UDP
        
        → 비연결형 프로토콜. 사전에 약속 없이 데이터 보냄. 받는 쪽에서 받았는지 확인도 안함. → 절차가 간단해서 속도가 빠르다.
        
        → 유튜브 라이브 스트리밍 하거나 게임할 때 사용.
        
- HTTP
    - 단순히 하이퍼 텍스트만 다루는게 아니라 웹 자원을 모두 포괄하는 전송 규약
    - 웹에서 데이터를 주고받기 위한 약속 → 브라우저와 서버가 대화하는 방법
    - 특징
        - 요청-응답 구조 → 반드시 클라이언트의 요청으로 시작.
            
            → 클라이언트가 서버에게 데이터 요청하면 서버는 해당 요청 처리하고 응답 반환
            
        - 무상태성(stateless) → 서버는 이전 요청 정보를 기억하지 않고 모든 요청을 독립적인 요청으로 처리
            
            → 장점 : 서버 구조 단순화, 유지보수 편의성, 서버 확장성 향상
            
            → 단점 : 로그인 상태 유지처럼 사용자 정보를 계속 기억해야 하는 기능이 필요.
            
            → 쿠키, 세션, JWT 토큰 사용
            
    - HTTP 버전별 차이
        - HTTP/1.1 - 제일 오랫동안 사용된 표준. 하나의 연결당 하나의 요청만 처리 → 느리다
        - HTTP/2 - 하나의 연결로 여러 요청을 동시에 처리(멀티 플렉싱) → 대부분 HTTP/2 사용
        - HTTP/3 - 기존 HTTP는 TCP 기반이었다면 HTTP/3는 UDP 기반 프로토콜(QUIC)
- HTTP - Request Method
    - Request Method를 통해서 요청의 의도를 명확하게 한다.
    - 웹 서버에게 어떤 작업을 수행하길 원하는지 알리는 방법을 정의하는 명령 규칙
    - 사용 이유 : 웹 서버에게 구체적인 작업을 명확하게 지시하기 위해 → 서버 작업 효율 높이고 요청의 의도를 명확하게 정의해서 문제 줄임.
    - 멱등성 : 같은 요청을 여러 번 보내도 결과가 똑같은 성질
        
        → 중요한 이유 : 같은 요청이 여러번 보내질 수 있다.
        
    - GET - 데이터 조회에서 이용
        
        → URL 입력하고 엔터 치면 GET 요청 보내는 것
        
        → 서버에 있는 특정 리소스 가져오기 위해 사용, 읽기 작업에만 사용
        
        → 먹등성을 가진다. → 한 번 조회하든 여러번 조회하든 결과는 똑같음.
        
    - POST - 리소스 전달,생성 → 회원가입 요청, 새로운 게시글 작성, 장바구니 상품 담기.
        
        → 멱등성을 보장하지 않는다.→ 회원가입 요청 두 번 보내면 2명의 사용자가 생김.
        
    - PUT - 서버로 데이터를 전송하여 리소스 전체 업데이트
        
        → 멱등성을 가진다. → 같은 내용으로 리소스를 한 번 교체하든 여러번 교체하든 최종 결과 동일
        
    - PATCH - 서버의 리소스를 부분적으로 수정 또는 업데이트
        
        → 멱등성을 보장하지 않는다. →보낼 때 마다 결과가 달라진다.
        
    - DELETE - 서버의 데이터를 삭제할 때 사용
        
        → 멱등성을 가진다.
        
        → 실무에서는 복구와 데이터 보존을 위해 Soft Delete를 많이 사용(is_deleted, deleted_at)
        
- HTTP - Status Code
    - 웹 통신 규약에서 요청의 처리 상태를 나타내는 코드
    - 사용 이유
        1. 원활한 소통을 위해 → 클라이언트와 서버는 HTTP 상태 코드를 통해 요청 결과를 서로 이해할 수 있음.
        2. 디버깅 효율 향상 → 상태 코드만 봐도 문제 원인을 빠르게 추측
    - 사용 방법 : HTTP 상태 코드를 응답 메시지의 헤더에 포함시켜 클라이언트에게 전송
    - 상태 코드
        
        → 1xx : 정보 메시지
        
        → 2xx : 성공
        
        → 3xx : 리다이렉션(서버가 클라이언트에게 “이 주소 말고 저 주소로 다시 요청”하라 알려주는 것)
        
        →  4xx : 클라이언트 오류
        
        → 5xx : 서버 오류
        
        → 200 : 요청 성공, 201 : 요청이 성공적으로 처리되고, 새로운 데이터가 생성
        
        → 400 : 잘못된 문법으로 서버가 요청을 이해할 수 없음
        
        → 401 : 인증되지 않은 접근
        
        → 403 : 인가되지 않은 접근
        
        → 404 : 요청받은 자원을 찾을 수 없음
        
        → 500 : 서버가 처리 방법을 알 수 없음
        
        → 501 : 요청 방법이 서버에서 지원되지 않아 처리할 수 없음
        
        → 503 : 서버가 요청을 처리할 준비가 되지 않았음
        
- Query string과 Path Variable
    - Query string : 클라이언트에서 웹 페이지에 대한 추가 정보를 서버에 전달하기 위해 사용되는 문자열
        
        → `/posts?category=tech` → `/posts` 만으로도 데이터를 조회할 수 있지만 `?category=tech` 가 붙으면 기술 카테고리에 있는 게시글만 출력되는 결과
        
    - 사용 이유 : 검색, 필터링, 정렬, 페이지네이션 등의 ****추가적인 정보를 ****클라이언트에서 서버에 제공하기 위해서
    - 여러 조건은 &로 연결해서 서버에 전달
    - Path Variable : 특정 자원을 식별하는 변수. URL 경로 그 자체에 포함되어서 자원의 정체성을 규정
        
        → /posts가 게시글 목록이라는 자원 /posts/123 → Path Variable은 123
        
    - 사용 이유 : 클라이언트에서 특정 리소스에 접근하기 위해
- API
    - 어플리케이션간 소통하기 위한 접점, 창구(Interface)
    - 서로 다른 시스템이나 애플리케이션 간에 데이터를 교환하고 기능을 공유하기 위해
    - 클라이언트 → API 요청 → 서버 처리 → API응답(상태코드로) → 클라이언트 결과 받아 화면 변경
    - 사용 이유
        
        → 클라이언트와 서버 역할을 나눌 수 있다.
        
        → 클라이언트는 서버 내부 코드나 DB 구조를 몰라도 됨.
        
        → 클라이언트와 서버가 안전하게 소통하기 위해 사용
        

## 헷갈린 점

- HTTPS 동작 방식이 헷갈렸다.
- 무상태성, 멱등성 같은 처음보는 개념이 헷갈렸다.

## 느낀 점

인터넷과 웹의 동작 원리, HTTP 통신 구조, API, TCP/IP 같은 네트워크 기초 개념들을 흐름 중심으로 이해할 수 있었다. 그리고 단순 암기보다 “ 왜 사용하는지”를 생각하며 학습하는게 중요할 거 같다.
