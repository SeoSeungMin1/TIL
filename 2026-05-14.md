# 2026-05-14(목)

## 배운 내용

- JVM 동작 과정
    1. 개발자가 .java 파일로 자바 소스 코드 작성
    2. 소스 코드를 javac 명령어를 통해 컴파일 → 결과물 : 바이트 코드(.class 파일)
    3. JVM은 클래스 로더를 사용해 필요한 .class 파일을 읽고 메서드 영역에 로드
    
    → 필요할 때만 클래스를 메모리에 올리는 지연 로딩 방식
    
    → 클래스 로딩 과정 (JVM이 실행 전에 사용하는 준비 기능)
    
    1. Loading : .class 파일을 찾아 JVM메모리의 메서드 영역에 올림
    2. Linking : 로딩된 클래스가 실제로 사용되기 전 올바르게 정의되어 있는지 확인하고 실행에 필요한 준비 작업 수행
        
        → 검증 : 보안과 안정성을 위해 클래스 파일의 구조가 정확히 작성되었는지 검사
        
        → 준비 : static 변수들을 위한 메모리를 JVM이 확보 / 변수들의 기본값(int = 0, boolean = false, 객체 = null)
        
        → 해결 : 심볼릭 레퍼런스(이름만 있는 상태) → 다이렉트 레퍼런스로 바꿈(실제 클래스의 메모리 주소를 가리키는 상태)
        
    3. Initializing : static 변수 실제 코드에 정의된 값으로 초기화
    1. 실행 준비 : 런타임 데이터 영역 구성
        
        → 메서드 영역 : 클래스 정보를 저장(클래스 설계도 역할), 런타임 상수 풀 구조 포함 → .class 파일 내의 상수 테이블을 JVM이 실행하는 형태로 변환
        
        → 힙 영역 : 실행 중 생성되는 모든 객체 인스턴스와 배열 저장. new로 생성된 객체 위치. JVM이 자동으로 메모리 할당하고, 사용 끝난 객체를 제거하는 Garbage Collection 대상.
        
        → 스택 영역 : 참조 변수가 위치하며 힙 객체의 주소값을 가지고 있음
        
        → PC 레지스터 : 현재 실행중인 바이트코드 명령어의 주소 저장
        
        → 네이티브 메서드 스택 : C/C++ 등의 외부 코드 실행 정보 저장
        
    2. Execution Engine(실행 엔진)
        
        → 클래스 로더에 의해 메서드 영역에 적재된 바이트코드를 실제로 실행
        
        → 실행 엔진 : 인터프리터 방식, JIT 컴파일 방식
        
        → 인터프리터 : 바이트 코드를 명령어 단위로 한 줄씩 읽으면서 실행. 장점 → 바로 실행 가능, 단점 → 실행 속도 느림
        
        → JIT컴파일러 : 자주 실행되는 코드를 감지하여 해당 바이트코드 전체를 한 번에 기계어로 컴파일.
        
        → 장점 : 실행 속도 빠름 , 단점 : 초기 실행 느림. JVM 구조 복잡
        
- Garbage Collection(GC)
    - 더 이상 사용되지 않는 객체를 자동으로 메모리에서 제거하는 기능
    - 힙 영역에 객체를 가리키는 참조가 없으면 GC가 자동으로 제거 → 힙 영역을 주기적으로 스캔
    - 사용 이유 : 개발자가 직접 메모리를 해제하지 않아도 되서 조금 더 비즈니스 로직에 집중할 수 있다.
- 힙 메모리 구조와 세대 분할
    - Young Generation : 객체가 처음으로 생성되는 영역
        
        → 수명 짧고 금방 소멸되는 객체가 주로 저장되어 GC 자주 발생
        
        → Minor GC 처리 대상 작고 실행 속도 빠름.
        
        → Eden은 객체가 처음으로 할당되는 공간, 살아 남으면 Survivor 영역으로 이동 → survivor 두 개 번갈아 사용, 일정 횟수 생존 → Old Generation으로 이동
        
    - Old Generation : 일정 시간 이상 살아남은 객체들이 저장되는 영역
        
        → 발생 빈도 낮고 실행 시간 오래걸림(Major GC)
        
        → Major GC는 천제 애플리케이션 실행 멈추는 Stop-The-World 현상 발생
        
- 클래스 : 속성(데이터)과 메서드(기능)를 묶어 객체를 생성하는 템플렛(설계도)
    - 사용이유
        
        → 코드의 재사용성 : 코드를 한번만 작성하여 재사용 가능
        
        → 유지보수성 : 변경사항이 발생했을 때 수정해야할 코드 최소화할 수 있고 관리 용이.
        
        → 모듈화 : 클래스는 관련된 속성과 메서드를 하나의 캡슐처럼 묶어준다.
        
        → 추상화 : 복잡한 내부 구현을 숨기고 사용자에게 필요한 기능을 노출
        
    - 객체 / 인스턴스 : 실제 생성된 대상 자체 / 어떤 클래스의 인스턴스(어떤 클래스 기반인지를 강조하는 차이)

## 헷갈린 점

- JVM 동작 과정 중 클래스 로더부터 이해하는데 용어가 헷갈렸다.
- 힙 영역을 대상으로 수행하는 GC 종류와 발생하는 현상들이 헷갈렸다.

## 느낀점

JVM을 깊게 파본적이 처음이라서 처음에 봤을 때는 단순히 바이트코드를 실행하는 가상머신 이렇게만 이해했었는데 깊게 파보니까 생각보다 이해가 안 가는 부분이 많아 시간 투자를 많이 해야겠다고 느꼈고, Garbage Collection도 단순히 객체를 가리키는 참조가 없으면 자동으로 제거해주는 것만으로 이해하고 있었는데 세대 분할로 어디에서 GC가 자주 발생하고 어디서는 발생 빈도가 낮은지 알게 되었는데 이것또한 시간투자를 많이 해야될 것 같다.
