## 배운 내용

- Thread
    - 하나의 프로세스 안에서 동시에 실행될 수 있는 독립적인 실행 흐름
    - 자바의 Thread : 운영체제와 JVM이 관리하는 실행 단위
    - 프로세스 vs 스레드
        
        → 프로세스 : 실행중인 프로그램 전체 단위
        
        → 스레드 : 그 프로세스 안에서 실제 일을 수행하는 흐름
        
    - 스레드도 자원이고 스레드마다 스택 메모리 필요.
    - 사용 이유 : 여러 작업을 동시에 실행해 애플리케이션의 성능과 응답성을 높이기 위해.
        
        → 성능 : 같은 시간 안에 더 많은 일을 처리하는 능력
        
        → 응답성 : 사용자가 요청했을 때 얼마나 빨리 반응하는가
        
        → 스레드 쓰면 무거운 작업을 분리해서 메인 흐름 덜 막히게 가능
        
    - 비동기 처리를 가능하게 만드는 기본적인 실행 흐름.
        
        → 동기 : 한 작업이 끝날 때 까지 다음 작업으로 넘어가지 않는 방식
        
        → 비동기 : 어떤 작업이 끝날 때 까지 전체 흐름을 멈추지 않고 그 작업은 따로 진행하게 만들고 현재 흐름은 알아서 다음 일을 하게 만드는 방식
        
    - 멀티스레드 환경 → 하나의 공유 데이터 사용하면 → 동시성/ 가시성 문제 발생 → 동기화 필요
        
        → 동시성 문제 : 여러 스레드가 동시에 같은 공유 데이터를 수정하거나 접근하면서 실행 순서가 꼬여 예상과 다른 결과가 나오는 문제
        
        → 해결 : Synchronized, Lock, CAS
        
        → 가시성 문제 : 한 스레드가 변경한 값을 다른 스레드가 바로 보지 못하는 문제.(같은 변수인데 스레드마다 값이 다르게 보이는 상황 발생)
        
        → 해결 : Volatile, Synchronized, Lock
        
        → 동기화 : 이런 문제를 막기 위해 여러 스레드가 공유 자원에 접근하는 순서를 조절하는 방법
        
    - 사용 방법 : Thread 클래스를 상속하거나, Runnable 인터페이스 구현 후 .strat() 메서드 호출해 실행
        
        → run() 호출하지 않고 strat() 호출하는 이유
        
        : run()을 직접 호출하면 별도의 실행 흐름이 생성되지 않고 현재 스레드가 실행되고, start()는 새로운 스레드를 생성한 뒤 내부적으로 run()을 실행하기 때문에 멀티스레드 환경을 만들기 위해.
        
    - 주요 메서드
        
        
        | start() | 새로운 스레드 생성, run() 실행 |
        | --- | --- |
        | run() | 스레드가 실제 수행할 작업 코드 |
        | sleep(long millis) | 현재 실행중인 스레드 잠시 멈춤 |
        | join() | 다른 스레드 작업이 끝날 때까지 기다림 |
        | interrupt() | 스레드에게 중단 요청 전달(강제 종료x) |
    - 스레드에서 발생한 예외는 보통 run() 내부의 try-catch로 처리
- Runnable
    - Runnable : 실행할 작업, Thread : 그 작업을 실행하는 주체
    - 스레드가 실행할 작업을 정의하기 위한 표준 인터페이스.
    - 사용 이유 : 실행 로직을 유연하게 정의하고, 다양한 비동기 환경에서 재사용 가능하게 하기 위해서
    - Runnable vs Callable
        
        
        | 비교 | Rnnable | Callable |
        | --- | --- | --- |
        | 메서드 | run() | call() |
        | 반환값  | x | o |
        | 예외 처리 | 제한적 → run() 안에서 직접 try-catch해야함.(내부에서 처리) | 가능→ throws Exception 가능(예외를 밖으로 던질 수 있음) |
    - 람다식 : 자바에서 간단한 메서드를 한 줄의 식으로 작성하는 문법
    
    ```jsx
    Runnable task = new Runnable() {
        @Override
        public void run() {
            System.out.println("작업 실행");
        }
    };
    ```
    
    ```java
    Runnable task = () -> System.out.println("작업 실행");
    ```
    
    - class MyTask extends Thread → 이 클래스는 다른 클래스 상속 불가
    - class MyTask impl Runnable → 인터페이스의 구현이여서 다른 클래스 상속 중에서 추가할 수 있다.
        
        → 실무에서 implements Runnable 방식 많이 사용.
        
    - run() 메서드 하나 가지며 “무엇을 할지” 정의
- Executor Service
    - 일종의 스레드 관리자
    - 스레드 풀 기반의 비동기 작업 실행을 관리하는 고수준 스레드 실행 서비스 인터페이스
        
        → 스레드 풀 기반 : 스레드를 매번 새로 만들지 않고 미리 만들어둔 스레드를 재사용
        
        → 비동기 작업 실행 : 작업 시켜놓고 결과 기다리지 않고 다른 일 먼저 할 수 있다.
        
        → 고수준 : 작업 실행, 스레드 생성, 재사용 자동으로 관리하고 종료 기능도 제공하는 실행 환경 제공
        
    - 사용 이유 : 스레드 자원 관리 자동화, 대량 작업을 안정적으로 병렬처리
    - Executor Service 구조(스레드 풀을 이용해서 작업을 실행하고 관리하는 객체)
        1. 작업 생성(정의)
            - Runnable
            - Callable
        2. Excutor Service에 작업 제출
            - execute()
            - submit()
        3. 작업이 BlockingQueue에 대기
        4. 스레드 풀의 유휴 스레드(현재 작업 실행 안하고 대기중인 스레드)가 작업을 가져감
        5. 스레드가 run() 또는 call() 실행
        6. 작업 완료 후 스레드는 종료되지 않고 다시 유휴 상태
        7. 사용이 끝나면 shutdown()으로 종료
    - Thread Pool : 미리 생성한 스레드를 보관하고, 작업이 들어오면 스레드가 재사용하여 처리
    - Executor Service 생성
        
        → ExecutorService executor = Executors.newFixedThreadPool(3)
        
    - execute() vs submit()
        
        
        | 비교 | execute() | submit() |
        | --- | --- | --- |
        | 반환값 | x | Future 반환 |
        | 결과 추적 | x | o |
        | Callable사용 | x | o |
    - Future 객체 : 나중에 결과 받을 수 있는 티켓 같은 객체
        
        → submit()은 작업 결과를 나중에 확인하거나 받을 수 있도록 Future 객체 반환
        
    - shutdown() : 일반적인 종료 처리, shutdownNow() : 예외 상황, 강제 종료 필요 시 사용
- Thread Pool
    - Executor Service : 작업 실행을 관리하는 고수준의 일종의 인터페이스
    - 실제 스레드 수와 큐, 거부 정책 세밀하게 다루는 대표 구현체가 스레드 풀(ThreadPoolExecutor)
    - 큐 사용 이유 : 순차 대기, 처리량 컨트롤
    - ThreadPoolExecutor : 스레드 풀을 직접 생성하고 제어할 수 있도록 설계된 클래스.
    - Executors.newFixedThreadPool(3)이면 corePoolSize = 3
        
        → 기본 스레드 3개 유지
        
        → corePoolSize보다 스레드 수가 작으면 새 스레드 생성해서 작업 실행
        
        → 이미 corePoolSize 만큼 스레드 사용 중이면 작업은 큐에서 대기
        
    - BlockingQueue에도 한계가 있음. maximumPoolSize까지 스레드를 추가로 생성
    - Executor Service 구조에서 순서 확인.

## 헷갈리는 점

- 멀티 스레드에 동시성 문제와 가시성 문제가 정확히 어떤건지 헷갈렸다.
- ExecutorService 내부에서 Thread Pool의 스레드가 작업을 실행하는 흐름과 corePoolSize가 어떤 의미를 가지는지 이해하는 부분이 조금 헷갈렸다.
