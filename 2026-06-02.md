## 배운 내용

- DI
    - Spring MVC + Controller-Service-Repository 요청 흐름
        1. 사용자가 브라우저나 스마트폰 앱 클릭
        2. 프론트엔드(react, vue)에서 클릭을 인식하고 인터넷을 통해 백엔드 서버로 HTTP Request를 쏴서 보낸다.
        3. Spring Boot로 도착하고, 작성해놓은 Controller로 온다.
        4. Controller는 요청 받고 해석하고 핵심 비즈니스 로직을 처리하는 Service클래스로 넘긴다.
        5. Service 클래스는 계산이랑 판단, 비즈니스 로직 처리하고 데이터를 영구적으로 저장하거나 조회하기 위해 Repository 계층에 지시를 내린다.
        6. Repository는 DB와 통신하기 위해 존재하는 계층
        7. Repository는 데이터베이스 접속해서 데이터를 기록하거나 조회를 한다.
        8. 처리가 끝나면 역순으로 다시 Service에게 전달하고 Controller에게 전달해서 프론트엔드에게 응답
        
        →  백엔드 서버 내부에서 컨트롤러는 서비스에게 서비스는 레포지토리에게 작업을 넘기고 주고 받으면서 진행
        
    - 객체가 직접 의존 객체를 생성하지 않고 ,외부에서 필요한 의존성을 주입받도록 하는 설계 방식
        - 의존
        
        → 어떤 객체가 책임을 하기 위해서 다른 객체의 기능을 필요로 하는 것
        
        → 자바 클래스와 다른 자바 클래스간의 논리적인 협력 관계
        
        - 주입
        
        → 필요한 객체를 클래스 내부에서 직접 생성하지 않고 외부(Spring 컨테이너)에서 전달받는 것
        
        → DI를 사용하지 않은 경우
        
        ```java
        public class OrderService {
        
            private OrderRepository repository = new OrderRepository();
        }
        ```
        
        → 주입 방식 사용
        
        ```java
        @Service
        public class OrderService {
        
            private final OrderRepository repository;
        
            public OrderService(OrderRepository repository) {
                this.repository = repository;
            }
        }
        ```
        
    - Spring boot에서 build.gradle : 환경설정 파일
    - DI 사용 이유
        - 결합도를 낮추기 위해
        
        → DI 사용 시 컴파일 시점에는 인터페이스에 의존, 런타임 시점에는 실제 구현체 주입
        
        → 코드상의 의존성과 실제 실행 시점의 의존성을 분리 → 객체 간 결합도를 낮추고, 구현체 교체와 유지보수가 쉬워짐
        
        - 테스트 용이성
        
        → 테스트 코드 작성할때 인터페이스로 잡아두고 가짜 객체를 넣어둬서 성공한다는 가정하에 생성자에 밀어넣어주는 역할
        
        → 네트워크 연결하지도 않고 비용도 들지 않고 안정적인 테스트를 만들수가 있다.
        
- Bean
    - Spring IoC 컨테이너가 생성하고 관리하는 재사용 가능한 Java 객체
    - Spring IoC 컨테이너 : 애플리케이션에서 사용되는 객체들을 대신 생성하고, 관리하고, 연결해주는 역할을 하는 핵심 구성 요소
    - Controller, Service, Sepository →Bean으로 등록
    - Bean은 객체 생성, 의존성 연결, 생명주기 관리를 Spring 컨테이너가 담당.
        
        → 개발자가 객체를 직접 생성하고 관리 안해서 비즈니스 로직 구현에 집중할 수 있고, 그 결과 의존성 주입(DI)을 통해 객체 간 결합도를 낮출 수 있다.
        
    - 어떤 객체를 Bean으로 등록하는가?
        
        → 모든 객체를 Bean으로 등록하는 것은 아님.
        
        → 주로 일회성 데이터 객체를 처리하는 핵심 처리 로직 클래스 등록
        
    - 사용 이유
        
        → 객체를 컨테이너가 관리하도록 표준화해, 의존성 관리·생명주기·구성 변경을 일관되게 처리하고 애플리케이션의 확장성과 안정성을 높이기 위해서
        
        1. 의존성 관리 자동화
            1. Controller는 Service에 의존하고, Service는 Repository, 결제 모듈, 알림 서비스 등에 의존할 수 있다.
            2. 만약 Spring Bean이 없다면 개발자가 직접 `new` 키워드로 객체를 생성하고 순서에 맞게 연결해야 한다.
            3. Spring 컨테이너는 이러한 복잡한 의존 관계를 분석하여 객체 생성과 연결을 자동으로 처리한다. → 개발자는 비즈니스 로직 구현에 집중
        2. 일관된 생명주기 관리
            1. 직접 객체를 생성하면 개발자가 초기화 시점과 순서를 모두 관리해야 한다.
            2. 컨테이너가 객체 생성, 초기화, 소멸 시점을 관리해주며 관련 기능도 제공한다. → 객체 생명주기를 일관되고 안정적으로 관리 가능
        3. 구성 변경 및 유연한 확장
            1. 개발 환경에서 가짜 결제 모듈을 사용하고, 운영 환경에서 실제 결제 모듈을 사용하는 경우
            2. Spring Bean 구조에서는 비즈니스 로직을 수정하지 않고 설정만 변경하여 다른 구현체를 주입
            3. Spring 컨테이너가 환경에 맞는 Bean을 자동으로 선택하여 주입
    - 사용 방법
        - 컴포넌트 스캔 기반 등록
            
            → 내 프로젝트 소스 코드 클래스 위에 `@Service` 같은 annotation 붙이면 가능
            
            | 어노테이션 | 설명 |
            | --- | --- |
            | `@Controller` | 웹 요청을 받아 처리하고, 주로 화면(View)을 반환하는 컨트롤러 역할의 Bean을 나타냅니다. |
            | `@RestController` | 웹 요청을 받아 처리하고, JSON과 같은 데이터(Response Body)를 반환하는 REST API용 컨트롤러 Bean을 나타냅니다. |
            | `@Service` | 비즈니스 로직과 정책을 담당하는 Service 계층의 Bean임을 명시합니다. |
            | `@Repository` | 데이터 접근 계층의 Bean임을 나타내며, 저장소 역할을 수행하고 데이터 관련 예외 처리를 포함합니다. |
            | `@Component` | 특정 계층 의미 없이, Spring 컨테이너가 관리해야 하는 일반적인 Bean임을 나타냅니다. |
            
            → DTO : JSON 데이터를 담기 위해 사용하는 자바 객체
            
        - 설정 클래스 기반 등록
            
            → 외부 라이브러리 클래스는 이미 컴파일된 상태로 제공되기 때문에 소스 코드를 수정할 수 없음.
            
            → Spring의 설정 클래스 `@Configuration` 와 Bean 등록 메소드 `@Bean` 사용하여 수동으로 Spring Bean에 등록
            
    - Bean Scope
        
        → 스프링 컨테이너가 빈을 만들 때 몇 개나 찍어낼지, 어디까지 공유해서 사용할지, 빈을 언제까지 살리고 소멸시킬지(생명주기)
        
        → Singleton Scope : Bean은 별도의 설정 없으면 기본적으로 Singleton Scope
        
        → 컨테이너 안에 해당 타입의 Bean 객체가 단 하나만 생성, 애플리케이션 전체에서 하나의 객체를 공유하여 사용
        
        → 객체 생성 비용 감소
        
        → 메모리 사용량 감소
        
        → GC 부담 감소
        
        → 성능 향상
        
        → 상태 데이터는 메서드 파라미터나 지역 변수로 관리하는 무상태 설계를 사용해야 한다.
        
        → 지역 변수는 스레드별 독립적인 Stack 메모리를 사용하므로 동시성 문제를 방지할 수 있다
        
- AOP
    - 핵심 비즈니스 로직과 공통 관심사를 분리해, 공통 기능을 코드 수정 없이 일관되게 적용하도록 하는 프로그래밍 방식
    - 공통 관심사(횡단 관심사) : 로그 출력, 보안 검사, 트랜잭션 처리, 성능 측정
    - 비즈니스 코드 자체를 수정하지 않고, 메서드 실행 전·후 또는 특정 조건을 만족하는 지점에 공통 동작을 삽입
    - 포인트컷 : AOP에서 공통 기능을 어디에 적용할지 정하는 규칙
    - 사용 이유 : 공통 기능을 한 곳에 모아 코드 중복을 줄이고, 비즈니스 로직을 수정하지 않고도 기능을 추가하거나 변경할 수 있기 때문
    - 사용 방법
        1. Aspect로 공통 기능 분리
        
        ```java
        @Aspect // 공통 기능을 모아둔 클래스임을 선언
        @Component // 스프링 빈으로 등록되어야 AOP가 동작
        public class LoggingAspect {
        		// 공통 로직과 적용 규칙이 들어갑니다
        }
        ```
        
        → Aspect : 공통 기능들을 하나로 묶어놓은 클래스
        
        1. pointcut으로 적용 대상 지정
        
        ```java
        // Pointcut : "어디에 적용할것인가?" 공통 로직은 따로 만들어뒀는데 이걸 어디서부터 어디까지 적용?
        // "com.example 패키지 밑에 있는 모든 Service클래스의 모든 메소드에 적용하겠다"
        @Pointcut("execution(* com.example..service..*(..))")
        public void serviceLayer() {
        
        }
        ```
        
        → `* com.example..service..*(..)` 공통 기능을 서비스 패키지 안에 있는 클래스에게만 적용해라.
        
        → 컨트롤러나 repository는 건드리지 말고 service만 건드려라 라고 하는 거다.
        
        1. 어드바이스로 실행 시점 정의
        
        ```java
        		// advice 언제? 무엇을 할것인가?
        		// @Around -> 메소드 실행 전후 모두를 감싸서 통제
        		// 아까 지정한 serviceLayer() 규칙에 맞는 곳에 이 행동을 적용하겠다
            @Around("serviceLayer()")
            public Object logExecutionTime(ProceedingJoinPoint joinPoint) throws Throwable {
        
                System.out.println("[LOG] 메서드 실행 시작: " + joinPoint.getSignature().getName());
                long start = System.currentTimeMillis();
        
                try {
        		        // 여기가 핵심. 
        		        // 실제 비즈니스 메소드를 호출하고 제어권을 넘깁니다.
                    return joinPoint.proceed();
        
                } finally {
                    long end = System.currentTimeMillis();
                    System.out.println("[LOG] 메서드 실행 종료. 걸린 시간: " + (end - start) + "ms");
                }
            }
        ```
        
        → 메소드를 시작하기 직전에 로그 찍을지(@Before), 메소드 실행 이후(@After), 실행 전 후 둘 다 찍을지(@Around) 정의
        
        → joinPoint.proceed();라는 코드를 기점으로 위쪽이 메소드 실행전, 아래쪽이 메소드 실행 후의 로직
        
    - 실제 AOP 동작 흐름
        1. 사용자 요청
            
            → 프론트엔드가 JSON 데이터를 HTTP Request Body에 담아서 보냄
            
        2. Spring이 DTO 생성
            
            → Spring이 내부적으로 Jackson을 사용해 JSON을 DTO 객체로 변환(계층과 계층 사이 데이터 전송하는 객체)
            
        3. DTO 검증(Bean Validation)
            
            → DTO에 검증 규칙이 선언 되있을 시 DTO 생성 직후 Bean Validation 수행
            
        4. Controller 진입
            
            → 검증 성공하면 Controller 진입
            
        5. Controller가 Service 호출
            
            → Controller 입장에서 `orderService.createOrder(request);`  호출했다고 생각
            
        6. 사실은 프록시 객체 호출
            
            → AOP가 적용되면 Controller가 주입받은 건 `OrderService`가 아니라 `OrderServiceProxy`
            
        7. 프록시가 호출 가로챔
            
            → 실제로 `OrderServiceProxy.createOrder()`가 먼저 실행
            
        8. Aspect 실행 → 프록시가 공통 기능 실행하는 단계
            
            → 진짜 Service를 실행하기 전 “로그 시작”을 출력하거나 공통 기능 수행
            
        9. 진짜 Service 실행
            
            → `joinPoint.proceed();` 실행 → `OrderService.createOrder()` 호출하는 코드
            
        10. 비즈니스 로직 수행
        11. OrderRepository → DB
        12. DB → OrderRepository
        13. 진짜 Service 끝남
        14. Aspect 마무리
            
            → 프록시가 다시 제어권 획득하여 “로그 종료” 또는 공통 기능 수행
            
        15. 응답 반환(Controller로 결과 반환)
            
            → Controller → Spring(Jackson) → JSON → 클라이언트
            
    - OOP vs AOP
        
        → OOP : 객체별 책임을 나누어 비즈니스 로직을 설계하는 것
        
        → AOP : 여러 객체에 공통 기능(로그, 트랜잭션, 권한검사 등)을 분리하여 관리하는 기술
        

## 헷갈린 점

실제 AOP 동작 흐름이 헷갈렸다.

## 느낀 점

DI, Bean, AOP의 개념과 동작 원리를 이해할 수 있었다. 특히 Spring이 객체를 관리하고 의존성을 주입하는 과정, 그리고 AOP가 프록시를 통해 공통 기능을 적용하는 흐름을 배우며 Spring의 내부 동작을 조금 더 깊이 이해할 수 있었다
