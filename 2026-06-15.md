## 배운 내용

- ORM
    - 자바 객체와 데이터베이스 테이블을 자동으로 매핑해 주는 기술
    - 객체와 테이블 사이의 구조적 차이(패러다임 불일치)를 해결하는 중간 계층
    - 개발자가 SQL보다 객체 중심으로 개발할 수 있도록 도와줌
    - 비즈니스 로직 구현에 집중할 수 있게 해줌
    - 사용 이유
        - 객체지향과 RDB의 패러다임 불일치 해결
        - 설계의 일관성과 코드의 생산성을 높이기 위해
    - 패러다임 불일치 예시
        - 상속 → 객체는 상속 가능, RDB는 상속 개념 없음
        - 연관관계 → 객체는 참조 사용(member.getTeam()), DB는 외래키 사용
        - 데이터 탐색 → 객체는 참조를 따라 탐색 가능, DB는 JOIN 결과 범위 내에서만 탐색 가능
        - 식별 방식 → 객체는 메모리 주소로 식별, DB는 PK로 식별
- JPA
    - 자바 객체와 관계형 데이터베이스를 매핑하고 객체 중심으로 데이터를 저장, 조회할 수 있게 해주는 ORM 표준 인터페이스
    - 실제 동작은 구현체가 담당하며, 대표적인 구현체 : Hibernate
    - JDBC를 직접 사용하지 않고도 객체를 DB에 저장하고 조회할 수 있도록 지원
    - 사용 이유
        - ORM 표준을 통해 안정성과 유지보수성 확보
        - 비즈니스 로직에 집중 가능
    - 1차 캐시
        - 같은 트랜잭션 내에서 조회한 엔티티를 메모리에 저장
        - 동일한 데이터를 다시 조회하면 DB가 아닌 1차 캐시에서 반환
        - DB 접근 횟수 감소
        - 객체 동일성 보장
    - JPA, Hibernate, JDBC 차이
        - JDBC : 자바와 DB를 연결하는 가장 기본적인 API
        - JPA : ORM 표준 인터페이스
        - Hibernate : JPA를 구현한 ORM 구현체
- Entity
    - 데이터베이스 테이블과 1:1로 매핑되는 자바 클래스
    - `@Entity`를 붙이면 JPA가 관리하는 대상이 됨
    - JPA는 엔티티를 영속성 컨텍스트에서 관리함
    - 사용 이유
        - 데이터와 비즈니스 규칙을 하나의 객체로 관리
        - 로직의 중복과 파편화 방지
        - 객체의 책임을 명확하게 분리
        - 응집도를 높여 유지보수성 향상
    - 특징
        - 데이터베이스 테이블과 매핑됨
        - JPA의 관리 대상
        - 영속성 컨텍스트에서 관리됨
        - 객체 스스로 자신의 상태를 관리할 수 있음
    - 설계 원칙
        - 기본 생성자 필수
            - 파라미터가 없는 기본 생성자가 반드시 필요
            - JPA는 리플렉션을 사용하여 객체를 생성함
            - 보통 `protected` 접근 제어자를 사용
        - final 사용 금지
            - 클래스에 final 사용 금지
            - 필드에 final 사용 금지
            - JPA는 지연 로딩을 위해 프록시 객체를 생성하는데, final 클래스는 상속이 불가능함
        - Setter 남용 금지
            - 클래스 레벨의 `@Setter` 사용 지양
            - 객체 상태가 어디서 변경되는지 추적하기 어려움
            - 객체의 일관성이 깨질 수 있음
            - 좋은 엔티티 설계 → Setter 대신 의미 있는 비즈니스 메서드 제공
            - 나쁜 엔티티 설계
                - user.setNickname(); → 왜 값이 변경되는지 의도를 알기 어려움, 변경 추적이 어려움, 객체의 책임 분산됨
- @Id
    - 엔티티의 기본 키(PK)를 지정하는 어노테이션
    - 데이터베이스의 PK와 자바 객체를 연결해주는 역할
    - 사용 이유
        - 엔티티를 유일하게 식별하기 위해
        - 영속성 컨텍스트가 엔티티를 관리하기 위해
        - 1차 캐시, 변경 감지(Dirty Checking) 등의 기능을 사용하기 위해
    - 기본 키 종류
        - 자연키(Natural Key)
            
            → 비즈니스 의미를 가지는 키
            
            → 이메일, 주민등록번호, 전화번호 등
            
            문제점 : 비즈니스 요구사항에 따라 변경될 수 있음, PK 변경 시 연관된 모든 FK 수정 필요, 데이터 정합성 문제 및 성능 저하 발생 가능
            
        - 대리키(Surrogate Key)
            
            → 식별만을 위해 인위적으로 생성한 키
            
            → AUTO_INCREMENT, SEQUENCE, UUID
            
            장점 : 비즈니스 정책 변경에 영향받지 않음, PK 변경이 거의 발생하지 않음
            
        
        → PK는 대리키 사용, 이메일, 전화번호 등은 UNIQUE 제약조건으로 관리
        
- 기본키 생성 전략
    - 엔티티의 PK 값을 자동 생성하는 방법
    - `@GeneratedValue`와 함께 사용
    - DB 환경과 성능 특성에 따라 전략 선택
    - TABLE 전략
        - 별도의 키 생성 테이블을 사용
        - 시퀀스 동작을 테이블로 흉내냄
        - 장점 : DB 종류에 관계없이 사용 가능, 이식성이 높음
        - 단점 : SELECT + UPDATE + INSERT 발생, 동시성 환경에서 락 발생 가능, 성능이 좋지 않음
        
        → 거의 사용하지 않음
        
    - SEQUENCE 전략
        - DB의 시퀀스 객체 사용
        - Oracle, PostgreSQL 등에서 사용
        - 동작 방식
            1. 시퀀스에서 PK 조회
            2. 엔티티에 PK 할당
            3. 영속성 컨텍스트 저장
            4. INSERT SQL은 쓰기 지연 저장소에 보관
            5. Commit 시 INSERT 실행
        - 장점 : 쓰기 지연 활용 가능, 성능 우수
        - allocationSize → 시퀀스 값을 한 번에 여러 개 가져오는 옵션
    - IDENTITY 전략
        - MySQL, MariaDB의 AUTO_INCREMENT 사용
        - 동작 방식
            1. persist() 호출
            2. JPA가 PK값이 없음을 확인
            3. DB가 PK를 생성해야 하므로 INSERT SQL 즉시 실행
            4. DB가 AUTO_INCREMENT 값 생성
            5. 생성된 PK 값을 JPA에게 반환
            6. JPA가 엔티티에 PK 할당
            7. 영속성 컨텍스트에 엔티티 저장
            8. 이후 트랜잭션 Commit 진행
        - 특징
            - DB가 PK 생성
            - INSERT를 해야 PK를 알 수 있음
            - 쓰기 지연 활용 어려움
        - 장점 : 설정이 간단하고 MySQL에서 가장 많이 사용
        - 단점 : 쓰기 지연과 충돌
    - AUTO 전략
        - JPA가 DB 환경에 맞는 전략 자동 선택
    - UUID
        - 전역적으로 유일한 문자열 기반 식별자
        - 장점 : 분산 환경(MSA)에서 충돌 없음, 쓰기 지연 사용 가능
        - 단점 : 문자열이라 크기가 크다, 인덱스 성능 저하 가능, 랜덤 값이라 클러스터드 인덱스 성능 저하 가능
- 영속성 컨텍스트
    - JPA가 엔티티를 관리하는 논리적인 메모리 저장소
    - 엔티티를 바로 DB에 반영하지 않고 영속성 컨텍스트에서 관리
    - EntityManager를 통해 접근하고 관리
    - 사용 이유
        - 객체와 데이터베이스 사이의 패러다임 불일치 해결
        - 변경사항 자동 추적
        - 트랜잭션 관리 편의성 제공
    - 주요 기능
        - 1차 캐시
            - 영속성 컨텍스트 내부의 캐시 저장소
            - PK를 Key, 엔티티를 Value로 저장
            - 동작 방식
            1. 엔티티 조회 요청
            2. 1차 캐시 확인
            3. 있으면 캐시 반환
            4. 없으면 DB 조회 후 캐시에 저장
            - 특징
                - 같은 트랜잭션 내에서 동일 엔티티 재사용
                - 객체 동일성 보장
                - 트랜잭션 범위에서만 동작
                - 전역 캐시가 아니다.
                - 성능 향상 효과는 제한적
        - 쓰기 지연
            - INSERT, UPDATE, DELETE SQL을 즉시 실행하지 않음
            - 쓰기 지연 SQL 저장소에 보관
            - 커밋 시점에 한 번에 실행
            - 장점
                - 네트워크 왕복 감소
                - 동시 처리 성능 향상
        - 변경 감지
            - 엔티티 조회 시 최초 상태를 스냅샷으로 저장
            - 커밋 시 현재 상태와 비교
            - 변경된 내용이 있으면 UPDATE SQL 자동 생성
            - 특징
                - 별도의 update() 메서드가 없음
                - 엔티티 값만 변경하면 자동 반영
    - flush
        - 영속성 컨텍스트의 변경 내용을 DB와 동기화하는 작업
        - 쓰기 지연 SQL 저장소의 SQL을 실제 DB에 전송
        - 트랜잭션은 아직 끝나지 않음
        - 실행 시점
            - 트랜잭션 Commit 직전 (자동)
            - `em.flush()` 직접 호출 (수동)
    - 엔티티 생명주기
        - 비영속 : 영속성 컨텍스트와 전혀 관계 없는 상태
            
            → `User user = new User();`
            
        - 영속 : 영속성 컨텍스트가 관리하는 상태
            
            → `em.persist(user);` 
            
            → 적용 가능 : 1차 캐시, 쓰기 지연, 변경 감지
            
        - 준영속 : 영속성 컨텍스트 관리 대상에서 분리된 상태
            
            → `em.detach(user);` `em.clear();` `em.close();`
            
            → 트랜잭션 종료, 영속성 컨텍스트 종료
            
        - 삭제 : 삭제가 예약된 상태
            
            → `em.remove(user);` 
            
            → Commit 시 DELETE SQL 실행
            
    - 동일성 vs 동등성
        - 동일성 : 같은 객체인지 비교, 메모리 주소 비교
        - 동등성 : 논리적으로 같은 값인지 비교
        - JPA 특징 → 1차 캐시를 통해 동일성 보장
- Transaction
    - 데이터베이스 상태를 변경하는 작업을 하나의 단위로 묶는 기능
    - JPA에서는 영속성 컨텍스트와 강하게 연결됨
    - 엔티티의 변경 사항을 추적하고 커밋 시점에 DB와 동기화
    - Commit 시 변경 내용을 반영하고, Rollback 시 변경 내용을 취소
    - 사용 이유
        - 데이터 일관성 보장
        - 여러 작업을 하나의 작업 단위로 처리
        - 오류 발생 시 전체 작업 복구 가능
        - JPA의 변경 감지, 쓰기 지연 기능 활용
    - JPA에서 역할
        - 영속성 컨텍스트의 생명주기와 연결됨
        - 엔티티의 상태 변화를 추적
        - 변경 사항을 DB에 안전하게 반영
        - 영속성 컨텍스트 기능이 동작하는 경계선 역할 수행
    - 동작 방식
        1. `@Transactional` 메서드 호출
        2. 트랜잭션 시작
        3. 영속성 컨텍스트 생성
        4. `em.find()` 또는 `em.persist()` 실행
        5. 엔티티 조회 또는 저장
        6. 엔티티 상태 변경
        7. 변경 내용 추적
        8. `commit()` 호출
        9. `flush()` 호출
        10. SQL 실행
        11. DB Commit
        12. 영속성 컨텍스트 종료
    - 조회 전용 트랜잭션 (`@Transactional(readOnly = true)`)
        - 조회만 수행하는 트랜잭션
        - 데이터 변경이 발생하지 않는다는 것을 JPA에 알려줌
        - 장점
            - 스냅샷 생성 생략
            - 변경 감지(Dirty Checking) 생략
            - 메모리 사용량 감소
            - 조회 성능 향상
    - OSIV
        - view(spring의 컨트롤러) 영역까지 session(영속성 컨텍스트)을 open 해둔다
        - Controller 및 View 영역에서도 지연 로딩 사용 가능
        - OSIV OFF
            - Controller → Service(@Transactional) → 트랜잭션 종료 → 영속성 컨텍스트 종료 → Controller
            - 문제점 : Service 종료 후 엔티티는 준영속 상태, 지연 로딩 수행 불가
        - OSIV ON
            - HTTP 요청 시작 → 영속성 컨텍스트 생성 → Controller → Service → Controller → 응답 → 영속성 컨텍스트 종료
            - 장점
                - Controller에서도 지연 로딩 가능
                - LazyInitializationException 방지
                - 개발 편의성 향상
            - 단점
                - DB 커넥션을 오래 점유
                - 외부 API 호출, 파일 업로드 중에도 커넥션 유지
                - 커넥션 풀 고갈 위험
                - 트래픽 증가 시 장애 가능성 증가
            - 해결방안 : OSIV 끄고 서비스 계층에서 필요한 데이터 페치 조인으로 가져오거나 지연 로딩 마치고 DTO로 변환해서 컨트롤러에 반환한다.
