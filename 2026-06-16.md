## 배운 내용

- ORM
    - 자바 객체와 데이터베이스 테이블을 자동으로 매핑해 주는 기술
    - 객체와 테이블 사이의 구조적 차이(패러다임 불일치)를 해결하는 중간 계층
    - 개발자가 SQL보다 객체 중심으로 개발할 수 있도록 도와줌
    - 비즈니스 로직 구현에 집중할 수 있게 해줌
    - 사용 이유
        - 객체지향과 RDB의 패러다임 불일치 해결
        - 설계의 일관성과 코드의 생산성을 높이기 위해
    - 패러다임 불일치 예시
        - 상속 → 객체는 상속 가능, RDB는 상속 개념 없음
        - 연관관계 → 객체는 참조 사용(member.getTeam()), DB는 외래키 사용
        - 데이터 탐색 → 객체는 참조를 따라 탐색 가능, DB는 JOIN 결과 범위 내에서만 탐색 가능
        - 식별 방식 → 객체는 메모리 주소로 식별, DB는 PK로 식별
- JPA
    - 자바 객체와 관계형 데이터베이스를 매핑하고 객체 중심으로 데이터를 저장, 조회할 수 있게 해주는 ORM 표준 인터페이스
    - 실제 동작은 구현체가 담당하며, 대표적인 구현체 : Hibernate
    - JDBC를 직접 사용하지 않고도 객체를 DB에 저장하고 조회할 수 있도록 지원
    - 사용 이유
        - ORM 표준을 통해 안정성과 유지보수성 확보
        - 비즈니스 로직에 집중 가능
    - 1차 캐시
        - 같은 트랜잭션 내에서 조회한 엔티티를 메모리에 저장
        - 동일한 데이터를 다시 조회하면 DB가 아닌 1차 캐시에서 반환
        - DB 접근 횟수 감소
        - 객체 동일성 보장
    - JPA, Hibernate, JDBC 차이
        - JDBC : 자바와 DB를 연결하는 가장 기본적인 API
        - JPA : ORM 표준 인터페이스
        - Hibernate : JPA를 구현한 ORM 구현체
- Entity
    - 데이터베이스 테이블과 1:1로 매핑되는 자바 클래스
    - `@Entity`를 붙이면 JPA가 관리하는 대상이 됨
    - JPA는 엔티티를 영속성 컨텍스트에서 관리함
    - 사용 이유
        - 데이터와 비즈니스 규칙을 하나의 객체로 관리
        - 로직의 중복과 파편화 방지
        - 객체의 책임을 명확하게 분리
        - 응집도를 높여 유지보수성 향상
    - 특징
        - 데이터베이스 테이블과 매핑됨
        - JPA의 관리 대상
        - 영속성 컨텍스트에서 관리됨
        - 객체 스스로 자신의 상태를 관리할 수 있음
    - 설계 원칙
        - 기본 생성자 필수
            - 파라미터가 없는 기본 생성자가 반드시 필요
            - JPA는 리플렉션을 사용하여 객체를 생성함
            - 보통 `protected` 접근 제어자를 사용
        - final 사용 금지
            - 클래스에 final 사용 금지
            - 필드에 final 사용 금지
            - JPA는 지연 로딩을 위해 프록시 객체를 생성하는데, final 클래스는 상속이 불가능함
        - Setter 남용 금지
            - 클래스 레벨의 `@Setter` 사용 지양
            - 객체 상태가 어디서 변경되는지 추적하기 어려움
            - 객체의 일관성이 깨질 수 있음
            - 좋은 엔티티 설계 → Setter 대신 의미 있는 비즈니스 메서드 제공
            - 나쁜 엔티티 설계
                - user.setNickname(); → 왜 값이 변경되는지 의도를 알기 어려움, 변경 추적이 어려움, 객체의 책임 분산됨
- @Id
    - 엔티티의 기본 키(PK)를 지정하는 어노테이션
    - 데이터베이스의 PK와 자바 객체를 연결해주는 역할
    - 사용 이유
        - 엔티티를 유일하게 식별하기 위해
        - 영속성 컨텍스트가 엔티티를 관리하기 위해
        - 1차 캐시, 변경 감지(Dirty Checking) 등의 기능을 사용하기 위해
    - 기본 키 종류
        - 자연키(Natural Key)
            
            → 비즈니스 의미를 가지는 키
            
            → 이메일, 주민등록번호, 전화번호 등
            
            문제점 : 비즈니스 요구사항에 따라 변경될 수 있음, PK 변경 시 연관된 모든 FK 수정 필요, 데이터 정합성 문제 및 성능 저하 발생 가능
            
        - 대리키(Surrogate Key)
            
            → 식별만을 위해 인위적으로 생성한 키
            
            → AUTO_INCREMENT, SEQUENCE, UUID
            
            장점 : 비즈니스 정책 변경에 영향받지 않음, PK 변경이 거의 발생하지 않음
            
        
        → PK는 대리키 사용, 이메일, 전화번호 등은 UNIQUE 제약조건으로 관리
        
- 기본키 생성 전략
    - 엔티티의 PK 값을 자동 생성하는 방법
    - `@GeneratedValue`와 함께 사용
    - DB 환경과 성능 특성에 따라 전략 선택
    - TABLE 전략
        - 별도의 키 생성 테이블을 사용
        - 시퀀스 동작을 테이블로 흉내냄
        - 장점 : DB 종류에 관계없이 사용 가능, 이식성이 높음
        - 단점 : SELECT + UPDATE + INSERT 발생, 동시성 환경에서 락 발생 가능, 성능이 좋지 않음
        
        → 거의 사용하지 않음
        
    - SEQUENCE 전략
        - DB의 시퀀스 객체 사용
        - Oracle, PostgreSQL 등에서 사용
        - 동작 방식
            1. 시퀀스에서 PK 조회
            2. 엔티티에 PK 할당
            3. 영속성 컨텍스트 저장
            4. INSERT SQL은 쓰기 지연 저장소에 보관
            5. Commit 시 INSERT 실행
        - 장점 : 쓰기 지연 활용 가능, 성능 우수
        - allocationSize → 시퀀스 값을 한 번에 여러 개 가져오는 옵션
    - IDENTITY 전략
        - MySQL, MariaDB의 AUTO_INCREMENT 사용
        - 동작 방식
            1. persist() 호출
            2. JPA가 PK값이 없음을 확인
            3. DB가 PK를 생성해야 하므로 INSERT SQL 즉시 실행
            4. DB가 AUTO_INCREMENT 값 생성
            5. 생성된 PK 값을 JPA에게 반환
            6. JPA가 엔티티에 PK 할당
            7. 영속성 컨텍스트에 엔티티 저장
            8. 이후 트랜잭션 Commit 진행
        - 특징
            - DB가 PK 생성
            - INSERT를 해야 PK를 알 수 있음
            - 쓰기 지연 활용 어려움
        - 장점 : 설정이 간단하고 MySQL에서 가장 많이 사용
        - 단점 : 쓰기 지연과 충돌
    - AUTO 전략
        - JPA가 DB 환경에 맞는 전략 자동 선택
    - UUID
        - 전역적으로 유일한 문자열 기반 식별자
        - 장점 : 분산 환경(MSA)에서 충돌 없음, 쓰기 지연 사용 가능
        - 단점 : 문자열이라 크기가 크다, 인덱스 성능 저하 가능, 랜덤 값이라 클러스터드 인덱스 성능 저하 가능
- 영속성 컨텍스트
    - JPA가 엔티티를 관리하는 논리적인 메모리 저장소
    - 엔티티를 바로 DB에 반영하지 않고 영속성 컨텍스트에서 관리
    - EntityManager를 통해 접근하고 관리
    - 사용 이유
        - 객체와 데이터베이스 사이의 패러다임 불일치 해결
        - 변경사항 자동 추적
        - 트랜잭션 관리 편의성 제공
    - 주요 기능
        - 1차 캐시
            - 영속성 컨텍스트 내부의 캐시 저장소
            - PK를 Key, 엔티티를 Value로 저장
            - 동작 방식
            1. 엔티티 조회 요청
            2. 1차 캐시 확인
            3. 있으면 캐시 반환
            4. 없으면 DB 조회 후 캐시에 저장
