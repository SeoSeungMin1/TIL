## 배운 내용

- Binary Search Tree
    - 이진 탐색 트리(Binary Search Tree)는 데이터를 효율적으로 탐색하기 위한 자료구조이다.
    - 규칙
        - 왼쪽 서브트리의 값은 부모 노드보다 작다.
        - 오른쪽 서브트리의 값은 부모 노드보다 크다.
        - 모든 서브트리도 BST 구조를 만족해야 한다.
    - 탐색 시 매 단계마다 검색 범위가 절반으로 줄어들어 평균 시간 복잡도는 O(log n)
    - 단점
        - 데이터가 정렬된 상태로 삽입되면 한쪽으로만 치우친 편향 트리가 생성될 수 있다.
        - 이 경우 탐색 성능이 O(n)까지 떨어질 수 있다.
- B-Tree
    - 데이터베이스와 파일 시스템에서 사용하는 균형 트리 구조이다.
    - 특징
        - 하나의 노드에 여러 개의 키를 저장할 수 있다.
        - 하나의 노드가 수백~수천 개의 자식을 가질 수 있다.
        - 트리 높이를 낮춰 디스크 접근 횟수(Disk I/O)를 줄인다.
    - 데이터베이스 성능은 디스크 I/O를 얼마나 줄이느냐가 중요하며, B-Tree는 이를 위해 설계
- B+Tree
    - 현재 대부분의 데이터베이스 인덱스가 사용하는 구조이다.
    - B-Tree와의 차이점
        - 실제 데이터는 리프 노드에만 저장된다.
        - 중간 노드는 탐색 경로 역할만 수행한다.
        - 리프 노드끼리 Linked List 형태로 연결되어 있다.
    - 장점
        - 중간 노드에 더 많은 키를 저장할 수 있어 높이가 더 낮아진다.
        - 범위 조회(Range Scan)에 매우 효율적이다.
        - 기간 조회, 가격 범위 조회, ID 구간 조회 등에 적합하다.
- Index
    - 데이터를 빠르게 조회하기 위한 자료구조이다.
    - 종류
        - 클러스터형 인덱스(Clustered Index)
            - 일반적으로 PK(Primary Key)
            - 데이터 자체가 정렬되어 저장된다.
        - 비클러스터형 인덱스(Non-Clustered Index)
            - 별도의 인덱스 페이지에 저장된다.
            - 개발자가 필요한 컬럼에 생성할 수 있다.
    - 인덱스 조회 과정
        1. 인덱스 페이지에서 키 검색
        2. 포인터 확인
        3. 실제 데이터 페이지 접근
    - 사용이유
        - 인덱스가 없으면 모든 데이터를 순차 탐색하는 Full Table Scan이 발생
        - 데이터가 많을수록 성능이 크게 저하
    - 단점
        - 추가 저장 공간 필요 (인덱스 페이지)
        - INSERT, UPDATE, DELETE 시 인덱스도 함께 갱신해야 한다.
        - 인덱스가 많을수록 쓰기 성능이 저하될 수 있다.
        - 두 단계(별도의 인덱스 페이지 만들기 → 실제 데이터 페이지 접근) → 인덱스를 탐색하고 데이터를 탐색하는 두 단계로 나뉘어져 시간이 걸림
- EXPLAIN
    - 데이터베이스가 SQL을 어떻게 실행할지 미리 보여주는 도구이다.
    
    | 항목 | 의미 |
    | --- | --- |
    | type | 데이터 접근 방식 |
    | possible_keys | 사용 가능한 인덱스 후보 |
    | key | 실제 사용한 인덱스 |
    | rows | 읽을 것으로 예상되는 데이터 수 |
    | Extra | 추가 실행 정보 |
    - type
        - ALL : 풀스캔
        - index : 인덱스 전체 스캔
        - range : 범위 검색
        - ref : 동등 비교 검색
        - const/system : PK 또는 Unique Key 단건 조회
    - Extra
        - Using where : WHERE 조건으로 필터링
        - Using index : 커버링 인덱스 사용
        - Using filesort : 별도 정렬 발생
        - Using temporary : 임시 테이블 사용
- 인덱스를 타지 못하는 경우
    - 컬럼 타입과 조건 타입이 다를 때
    - 컬럼에 함수 사용 시 → `DATE(created_at)`
    - `%검색어%` 형태의 LIKE 검색 사용 시
    - B+Tree 인덱스는 문자열의 앞부분 기준으로 정렬되므로 앞에 `%`가 붙으면 인덱스를 활용하기 어려움
- 카디널리티와 셀렉티비티
    - 카디널리티 : 컬럼에 저장된 값의 고유성 → 중복되는 값이 얼마나 적은가
        - 카디널리티 높음 : 주민등록번호, 이메일 주소, 주문번호 → 값이 중복되지 않는 것
        - 카디널리티 낮음 : 성별(남,여), 상태(활성, 비활성) → 값의 종류가 몇 개 안되서 중복이 매우 많음
    - 셀렉티비티 : 카디널리티 / 전체 행 개수로 계산하는 값. → 1에 가까울수록 좋음
    
    → 인덱스는 카디널리티가 높은 컬럼에 생성하는 것이 효과적이다.
    
- 복합 인덱스
    - 여러 컬럼을 묶어 하나의 인덱스를 생성하는 방식
    
    ```sql
    CREATE INDEX idx_location_first ON users(Location, Name, ID);
    ```
    
    ```sql
    //사용 가능
    WHERE Location='서울'
    WHERE Location='서울' AND Name='토르'
    WHERE Location='서울' AND Name='토르' AND ID=123
    ```
    
    ```sql
    //인덱스를 타지 못함
    WHERE Name='김철수'
    WHERE ID=123
    ```
    
    → 복합 인덱스는 왼쪽부터 사용
    
- 커버링 인덱스
    - 조회에 필요한 모든 컬럼이 인덱스 안에 포함된 경우
    - 실제 데이터가 있는곳까지 찾아갈 필요 없이 인덱스 페이지만 읽고 결과를 반환할 수 있다.
    - 장점
        - 조회 성능 향상
        - 디스크 접근 감소
- Full-Text Index
    - 대량의 텍스트 검색을 위해 사용하는 인덱스 방식
    - 동작 과정
        1. 토큰화
        2. 불용어 제거
        3. 역인덱스 생성
    - 역인덱스
        - 단어 기준으로 어떤 문서(Row)에 존재하는지 저장한다.
        - 검색 시 전체 테이블 탐색 없이 결과를 찾을 수 있다.
    - Full-Text Index 사용 시기
        - 검색 기능이 부가 기능일 때
        - 복잡한 검색 랭킹이 필요 없을 때
        - 검색량이 많지 않을 때
    - 검색 엔진 사용 시기
        - 검색이 핵심 서비스일 때
        - 대규모 트래픽 처리 필요
        - 고급 랭킹 및 필터링 필요
        - 확장성이 중요할 때
- Transaction
    - 데이터베이스에서 절대로 쪼개져서는 안 되는 논리적 작업 단위
    - 사용 이유
        - 데이터 무결성 보장
        - 데이터 일관성 유지
