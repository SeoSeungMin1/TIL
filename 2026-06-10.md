## 배운 내용

- Transaction
    - 여러 개의 DB 작업을 하나의 논리적인 작업 단위로 묶는 것
    - 예시 → 출금, 입금 둘 다 성공하거나 둘 다 실패해야 함.
    - 사용 이유
        
        → 데이터의 무결성과 일관성을 보장하기 위해
        
    - Atomicity(원자성)
        - 트랜잭션에 포함된 작업은 전부 성공하거나 전부 취소
        - 예시 → 출금 성공, 입금 실패 x → 전부 롤백
        - 부분적으로 성공하는 상태는 절대 허용하지 않음
    - Consistency(일관성, 정합성)
        - 트랜잭션이 실행을 성공적으로 완료하면 일관성 있는 DB 상태를 유지
        - 예시 → 계좌의 잔고는 0원 이상이여야한다 라는 규칙 정의
        - 규칙을 깨는 트랜잭션은 허용되지 않음
    - Isolation(격리성)
        - 동시에 실행되는 트랜잭션이 서로 영향을 주면 안된다.
        - 예시 → 재고가 1개인 상황에서 A,B 구매 동시에 처리하면 재고 -1 가능 → 동시성 문제를 막기 위해 트랜잭션 간 격리 필요
        - 동시성을 챙기기 위해 엄격하게 격리하면 처리 성능 떨어진다. → 격리 수준이라는 옵션 제공
    - Durability(지속성)
        - 커밋된 데이터는 장애가 나도 보존 → 영구적으로 기록
        - 예시 → 주문 완료 후 서버 다운되도 주문 데이터는 남아있어야 함
    - Isolation Level(격리 수준)
        - 격리 수준이 낮을수록 동시성 높아진다. → 처리가 빠르게 되서 성능이 좋아질 수 있고 정합성이 떨어짐
        - 격리 수준이 높을수록 동시성이 낮아진다. → 처리가 늦게 되고 병목 현상 발생하는 대신 정합성이 좋아짐
            
            → 격리 수준 높인다고 동시성 문제가 다 해결되지는 않는다.
            
        - Read Uncommitted(레벨0)
            - 커밋되지 않은 데이터도 읽을 수 있음
            - 가장 낮은 격리 수준
            - 발생 가능 : Dirty Read, Non-Repeatable Read, Phantom Read
            - 거의 사용 안 함
            - 데이터 정합성이 너무 쉽게 깨짐
        - Read Committed(레벨1)
            - 커밋된 데이터만 읽을 수 있음
            - 대부분의 RDBMS 기본 격리 수준
            - 발생 가능 : Non-Repeatable Read, Phantom Read
            - Dirty Read 방지
            - 적절한 성능과 정합성 제공
        - Repeatable Read(레벨2)
            - 같은 트랜잭션 안에서는 같은 데이터를 여러 번 조회해도 결과가 변하지 않음
            - 처음 조회한 시점의 스냅샷 유지
            - 발생 기능 :  Phantom Read
            - Dirty Read, Non-Repeatable Read 방지
            - 사용 : 보고서 생성, 통계 계산, 여러 단계 조회 후 계산하는 작업
        - Serializable
            - 트랜잭션을 순차적으로 하나씩 실행하는 것처럼 처리
            - 가장 높은 격리 수준
            - 데이터 정합성이 좋아짐
            - 단점 : 성능 저하, 병목 발생, 동시성 가장 낮음
            - 사용 : 금융 시스템, 재고 시스템, 정합성이 중요한 서비스
    - 이상 현상
        - 격리 수준을 어떻게 설정하느냐에 따라서 생길 수 있는 문제 현상
        - Dirty Read : 다른 트랜잭션이 Commit하지 않은 데이터 조회
        - Non-Repeatable Read : 같은 데이터 두 번 조회했는데 값이 달라지는 현상
        - Phantom Read : 같은 조건으로 조회했는데 데이터 개수가 달라짐
- NoSQL
    - Not only SQL → SQL만을 사용하지 않는 여러 유형의 DB
    - RDBMS 외의 다양한 데이터 모델 사용
    - RDBMS는 정해진 스키마, 테이블 구조에 맞춰서 데이터 저장
    - NoSQL은 엄격한 규칙에서 벗어나 유연한 데이터 구조로 처리
    - 사용 이유 : 유연한 구조, 대규모 데이터, 수평 확장
        
        → 수직 확장 : 기존 서버 사양 자체를 올리는 것, CPU, RAM 좋게 업그레이드.
        
        → 수평 확장 : 기존 서버와 비슷한 사양의 서버 여러대 추가해서 데이터 나누고 요청 분산하는 방식
        
    - 동작 방식
        - Key-Value
            - key → value 구조
            - HashMap을 데이터베이스로 확장한 느낌
            - 대표 : Redis
            - 사용 : 캐시, 세션, 인증코드, 카운트 처리하는 문제
        - Document DB
            - JSON 형태 저장
            - 객체 단위 조회
            - 기준 : “자주 함께 읽는 데이터를 하나의 문서로 묶을 수 있는가?”
            - 대표 : MongoDB
            - 장점 : JOIN 감소, 객체 단위 조회
        - 컬럼 DB
            - 분석하거나 대규모 집계
            - 큰 단위의 데이터 빠르게 찾기 위해서 사용하고 같은 row안에서 필요한 컬럼 묶음을 읽어서 처리
            - 사용 : 대규모 로그, 사용자 행동 이벤트, 시계열 데이터 → 쓰기량이 많고 대용량 처리가 필요할 때
        - Graph DB
            - 데이터간의 관계에 초점을 맞춘 DB
            - 노드랑 간선으로 표현
            - 대표 : Neo4j
            - 사용 : 친구 추천, 상품 추천, 지식 그래프
- 데이터베이스 확장
    - 확장 이유 : 성능 좋게 만들기 위해서, 안정적으로 처리되게 하기 위해서
    - 데이터베이스를 물리적 또는 논리적으로 분산시키는 기술
        
        → 분산 : 한 DB가 모든 데이터를 들고 모든 요청을 처리하게 두지 않고 데이터를 나누거나 복사해서 여러 DB가 역할을 나눠갖게 만든다
        
    - 스케일업(수직 확장) : 서버(DB) 한대의 성능을 업그레이드
    - 스케일 아웃(수평 확장) : 여러대의 서버(DB)가 일을 나눠 처리하게 하는것
    - 사용 방법
        - 파티셔닝
            - 데이터를 조각으로 나누는 것
            - 수직 파티셔닝 : 컬럼 기준 분리
                
                → 장점 : 조회 가볍고, 민감정보 분리
                
                → 단점 : JOIN 필요
                
            - 수평 파티셔닝 : Row 기준 분리
                
                → 장점 : 필요 데이터만 조회 가능, 인덱스 크기 감소, 쓰기 부하 감소
                
                → 단점 : 기준을 잘못 나누면 효과 없음, 전체 조회하면 모든 파티션 조회, 파티션 키 중요
                
        - 샤딩
            - 수평 파티셔닝된 데이터를 여러 DB나 노드에 분산해서 운영하는 방식
            - Range-based Sharding (범위 기반 샤딩)
                - 샤딩 키의 범위를 기준으로 데이터 분산
                - ID, 날짜 범위 등을 기준으로 나눔
                - 장점 : 구조가 단순하고 이해하기 쉬움, 범위 조회에 유리
                - 단점 : 데이터가 균등하게 분산되지 않을 수 있음, 다른 기준 조회에 약함
            - Hash-based Sharding (해시 긱반 샤딩)
                - 샤딩 키(user_id)를 해시 함수에 넣어서 나온 결과값을 기준으로 데이터를 분산
                - 장점 : 데이터가 고르게 분산, 특정 샤드에 부하 집중 방지, 쓰기 부하 분산 효과 좋음
                - 단점 : 범위 조회에 매우 불리 → 모든 샤드를 조회해야 할 수 있음, 데이터 위치를 직관적으로 알기 어려움
            - Composite Sharding (복합 샤딩)
                - 범위 기반 + 해시 기반 혼합
                - 장점 : 범위 조회 가능, 쓰기 부하도 분산 가능
                - 단점 : 설계가 복잡하고 운영 난이도 증가
        - Replication(복제)
            - 같은 데이터를 여러 서버에 복사
            - 사용 이유
                - 고가용성 : DB에서 장애 발생했을때 중단 시간 줄이기 위해
                - 부하분산 : 대부분의 서비스들에서 쓰기 작업보다 읽기 작업이 많이 일어남, 복제를 통해 읽기 부하 분산 → primary는 쓰기 작업에 집중하고 나머지 읽기 성능이 좋아짐.
            - Primary : 쓰기 담당, Replica : Primary 데이터를 복사
            - 동작 방식
                - Primary - 데이터 변경 발생
                - 변경 내용 Binary Log에 기록 → 시간 순서대로 저장 (작업 일지)
                - Replica - Primary의 Binary Log 확인해서 새로운 로그 생기면 자신의 DB에 동일하게 반영
                - 모든 Replica는 Primary 상태 따라감
            - 복제 지연 (Replication Lag)
                - Primary 변경 사항이 Replica에 반영되기까지 발생하는 시간 차이
                - 발생 원인 : 네트워크 지연, Replica 서버 부하, 로그 반영 속도 차이
                - 문제점 : 최신 데이터 조회 실패 → 실제로 가입됐는데 아직 복제가 안된 상태
                - 해결 전략
                    1. 중요한 조회는 Primary 사용해서 즉시 조회
                    2. 일정 시간 Primary 사용 → 복제 끝날 시간 확보
                    3. 복제 지연 모니터링
