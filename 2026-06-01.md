## 배운 내용

- 객체지향(OOP)
    - 객체지향은 현실 세계의 객체처럼 데이터와 기능을 하나로 묶어 설계하는 프로그래밍 패러다임입니다.
    - 핵심은 단순히 클래스에 변수를 넣는 것이 아니라,
        
        > "이 객체는 어떤 책임을 가져야 하는가?"
        > 
        > 
        > "다른 객체와 어떻게 협력하는가?"
        > 
    - 객체들은 서로 메시지를 주고받으며 협력하고, 내부 구현은 숨긴 채 필요한 기능만 제공하는데 이것을 캡슐화라고 합니다.
    - 사용이유 : 변경이 발생했을 때 영향 범위를 최소화하고 유지보수와 확장을 쉽게 하기 위해서
- 의존성(Dependency)
    - 객체지향에서는 객체들이 협력하기 때문에 의존성이 생깁니다.
    - 예를 들어 주문 서비스를 처리하려면 결제 서비스가 필요합니다.
    - 이때 OrderService는 PaymentService에 의존한다고 말합니다.
    - 의존한다는 것은 단순히 사용한다는 의미가 아니라,
        
        > "상대방이 변경되면 나도 영향을 받을 수 있다"
        > 
    - 의존성을 없애는 것이 아니라 적절하게 관리하는 것이 중요
- IoC
    - 예전에는 개발자가 직접 new를 사용해서 객체를 생성
        
        → 하지만 이렇게 되면 특정 구현체에 강하게 결합
        
    - Spring에서는 필요한 객체를 직접 생성하지 않고 주입받습니다. 이를 DI(의존성 주입)
    - 객체 생성과 관리 권한을 개발자가 아닌 Spring 컨테이너가 담당하는데 이것을 IoC(제어의 역전)라고 합니다.
        
        → 덕분에 개발자는 객체 생성보다 비즈니스 로직에 집중할 수 있습니다.
        
- MVC
    - MVC는 애플리케이션을 역할별로 분리하는 아키텍처 패턴
    - Model : 데이터와 비즈니스 로직
    - View : 사용자 화면
    - Controller : 요청 처리 및 흐름 제어
    - 예를 들어 레스토랑으로 비유하면
        - 손님 → 사용자
        - 웨이터 → Controller
        - 주방 → Model
        - 플레이팅된 음식 → View
    - 이렇게 분리하면 화면이 바뀌어도 비즈니스 로직은 영향을 받지 않아 유지보수가 쉬워집니다.
- Spring MVC
    - Spring MVC는 MVC 패턴을 웹 환경에서 구현한 프레임워크입니다.
    - 동작 순서
        - 요청이 들어오면
        1. DispatcherServlet 디스패처 서블릿이 요청을 받음
        2. HandlerMapping이 컨트롤러 탐색
        3. Controller 실행
        4. Service 호출
        5. 결과를 JSON 또는 HTML로 변환
        6. 클라이언트에게 응답
        7. ViewResolver 리솔버 (템플릿 엔진 사용할 때)
        8. View 렌더링 (템플릿 엔진 사용할 때)
            
            → 구조로 동작합니다. → 동작 과정 중요!
            
    - 에러 코드
        - 404 : 요청 주소와 연결된 컨트롤러 찾을 수 없음
        - 400 : 요청 데이터 변환 실패
        - 500 : 서버 내부 로직 오류

## 헷갈리는 점

- IoC와 DI의 차이가 둘 다 객체를 주입받는 개념이라고 생각해서 헷갈렸다.

## 느낀 점

- 객체지향은 단순히 클래스를 만드는 것이 아니라 객체의 책임과 협력을 중심으로 설계하는 것이 중요하다는 것을 알게 되었다.
- Spring MVC의 전체 요청 처리 흐름을 이해할 수 있었고, 앞으로 API를 개발할 때 내부 동작을 이해할 때 중요하다고 생각한다.
