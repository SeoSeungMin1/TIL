## 배운 내용

- 테스트 코드
    - 소프트웨어의 기능이 의도한 대로 동작하는지 검증하기 위해 작성하는 코드
    - 사람이 직접 확인하는 대신 코드가 자동으로 결과를 검증한다
    - 사용 이유
        - 버그를 조기에 발견할 수 있다.
        - 리팩토링 시 기능이 깨졌는지 확인하는 안전망 역할을 한다.
        - 유지보수 비용을 줄일 수 있다.
        - 기능의 동작 방식을 설명하는 살아있는 문서 역할을 한다.
        - 테스트하기 쉬운 코드를 작성하려다 보면 자연스럽게 좋은 구조의 코드가 만들어진다.
    - 장점
        1. 안전망 역할
            - 개발 중 예상하지 못한 문제를 빠르게 발견할 수 있다.
            - 테스트 실패 지점을 통해 버그 위치를 빠르게 찾을 수 있다.
        2. 리팩토링에 대한 자신감
            - 코드 구조를 변경해도 테스트가 모두 통과하면 기존 기능이 유지된다는 확신을 얻을 수 있다.
            - 변경에 대한 두려움을 줄여준다.
        3. 문서 역할
            - 기능이 어떤 결과를 의도하는지 가장 정확하게 보여준다.
            - 일반 문서와 달리 기능이 변경되면 테스트도 함께 수정해야 하므로 문서와 실제 동작이 어긋나기 어렵다.
        4. 좋은 설계 유도
            - 테스트하기 어려운 코드는 보통 책임이 많고 복잡하다.
            - 테스트를 작성하다 보면 기능을 분리하고 책임을 명확하게 나누게 된다.
    - 동작 방식
        1. 준비(Given/Arrange)
            - 테스트에 필요한 객체와 데이터를 준비한다.
            
            ```java
            Calculator calculator = new Calculator();
            int a = 5;
            int b = 3;
            int expectedResult = 8;
            ```
            
        2. 실행(When/Act)
            - 검증할 기능을 실행한다.
            
            ```java
            int actualResult = calculator.add(a, b);
            ```
            
        3. 검증(Then/Assert)
            - 실행 결과가 기대한 값과 같은지 확인한다.
            
            ```java
            Assertions.assertEquals(expectedResult, actualResult);
            ```
            
    - JUnit의 역할
        - 테스트 메서드를 자동으로 실행한다.
        - 결과를 비교하고 성공/실패를 판정한다.
        - 여러 테스트를 한 번에 실행할 수 있다.
    - 주의사항
        - 테스트 하나가 성공했다고 모든 상황이 검증된 것은 아니다.
        - 다양한 입력값과 예외 상황에 대한 테스트를 추가로 작성해야 한다.
- 좋은 테스트의 기준
    - 좋은 테스트란 버그를 안정적으로 찾아내고, 코드 변경에 대한 확신을 주며, 유지보수가 쉬운 테스트이다.
    - 단순히 테스트를 많이 작성하는 것이 아니라 좋은 기준을 만족해야 한다.
    - 대표적인 기준이 FIRST 원칙
    - FIRST 원칙
        - F - Fast (빠른 테스트)
            - 테스트는 빠르게 실행되어야 한다.
            - 개발자가 부담 없이 자주 실행할 수 있어야 한다.
            - 테스트가 느리면 실행을 꺼리게 되고 피드백이 늦어진다.
            
            → 느려지는 원인 : 실제 DB 사용, 외부 API 호출, 네트워크 의존
            
            → 해결 방법 : Fake 객체, Mock 객체 사용, 외부 의존성 제거
            
            → 테스트는 성능 자체보다 “자주 실행할 수 있는 속도”가 중요
            
        - I - Independent (독립적인 테스트)
            - 테스트끼리 서로 영향을 주면 안 된다.
            - 실행 순서와 상관없이 항상 같은 결과가 나와야 한다.
            
            → 나쁜 예 : testA가 객체 생성 → testB가 그 객체 사용 (testA가 실패하면 testB도 실패)
            
            → 좋은 방법 : 각 테스트마다 새로운 객체 생성, @BeforeEach 활용
            
            → 하나의 테스트가 실패하면 해당 시나리오만 실패해야 한다.
            
        - R - Repeatable (반복 가능한 테스트)
            - 언제 어디서 실행하든 같은 결과가 나와야 한다.
            
            → 반복 가능성을 깨뜨리는 요소 : 현재 시간, 랜덤 값, 네트워크 상태, 외부 시스템 상태
            
            → 좋은 방법 : LocalData.of(2025, 10, 11); 처럼 값을 고정하여 사용
            
            → Flaky Test : 어제는 성공, 오늘은 실패처럼 실행할 때마다 결과가 달라지는 테스트
            
            → 테스트 결과는 환경이나 시간에 영향을 받으면 안 된다.
            
        - S - Self-Validating (스스로 검증)
            - 사람이 결과를 확인하지 않아야 한다.
            - 테스트가 성공/실패를 스스로 판단해야 한다.
            
            → 나쁜 예 : System.out.println(result); 사람이 직접 확인해야 함
            
            → 좋은 예 : assertEquals(5, result); 테스트가 자동으로 판정
            
            → 테스트는 사람이 아니라 컴퓨터가 자동으로 판단해야 한다.
            
        - T - Timely (적절한 시기에 작성)
            - 테스트는 가능한 빨리 작성해야 한다.
            - 기능 개발 후 나중에 작성하는 것이 아니라 기능과 함께 작성하는 것이 좋다.
            - 이유
                - 나중에 작성하면 테스트하기 어려운 구조가 이미 굳어질 수 있음
                - 테스트 가능한 설계를 자연스럽게 유도
            
            → 테스트 작성을 미루지 말고 개발 과정에 포함시켜야 한다.
            
    - 테스트 커버리지
        - 테스트 코드가 프로덕션 코드의 몇 %를 실행했는지 나타내는 지표
        - 커버리지가 높다고 좋은 테스트는 아니다.
        - 검증 없이 코드만 실행해도 커버리지는 올라갈 수 있다.
        
        → 커버리지는 참고 지표일 뿐, 테스트 품질을 보장하지 않는다.
        
    - TDD
        - 테스트 주도 개발
        - 기능 코드보다 테스트를 먼저 작성하는 개발 방식
        - 개발 순서 : 테스트 작성 → 기능 구현 → 리팩토링
        - 단계
            1. Red - 실패하는 테스트 작성(무엇을 만들 것인지 먼저 정의)
            2. Green - 테스트를 통과할 최소한의 코드 작성(필요한 만큼만 구현)
            3. Refactor - 기능은 유지하면서 구조 개선(테스트가 안전망 역할 수행)
        - 장점
            - 요구사항을 먼저 생각하게 된다.
            - 결합도가 낮고 응집도가 높은 구조를 만들게 된다.
            - 리팩토링이 쉬워진다.
            - 테스트 가능한 설계를 유도한다.
