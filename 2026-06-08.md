## 배운 내용

- 데이터 베이스(Database)
    - 단순히 데이터를 저장하는 공간이 아니라 데이터를 체계적으로 저장하고 관리하여 의미 있는 정보로 활용할 수 있도록 하는 시스템
    - 사용 이유
        
        → 데이터 영속성 보장, 데이터 무결성과 신뢰성 보장, 여러 사용자의 동시 접근 지원, 데이터를 체계적으로 저장하고 관리하기 위해
        
    - RDB : 데이터를 테이블 단위로 저장하는 데이터베이스
        - 기본키(PK)
            
            → 각 데이터를 구별하는 유일한 값
            
            → 중복 불가, NULL 불가
            
        - 외래키(FK)
            
            → 다른 테이블의 PK를 참조하는 값
            
            → 테이블 간 관계를 형성하는 역할
            
        - PK와 FK를 통해 데이터 간 관계를 표현
    - 스키마
        - 테이블 구조와 데이터 규칙을 정의
        - 데이터 타입, NOT NULL, UNIQUE, FOREIGN KEY 등의 제약 조건 포함
        - 애플리케이션과 데이터베이스 사이의 계약 역할
        - 데이터베이스의 마지막 방어선
- 정규화
    - 데이터 중복을 최소화하고 무결성을 유지하기 위한 과정
    - 사용 이유
        
        → 데이터 이상 현상 방지
        
        → 데이터 구조를 효율적으로 관리하기 위해
        
    - 이상 현상
        - 삽입 이상 : 원하는 데이터를 입력하기 위해 불필요한 정보까지 함께 입력해야 하는 문제
        - 갱신 이상 : 중복된 데이터 수정 시 일부만 수정되어 데이터 불일치가 발생하는 문제
        - 삭제 이상 : 특정 데이터를 삭제했을 때 필요한 정보까지 함께 삭제되는 문제
    - 제 1정규형(1NF)
        
        → 모든 컬럼은 원자값을 가져야 함
        
        → 하나의 컬럼에는 하나의 값만 저장
        
        → 여러 값이 필요하면 별도 테이블로 분리
        
    - 제 2정규형(2NF)
        
        → 1NF 만족하면서
        
        → 부분 함수 종속 제거
        
        → 일반 컬럼은 기본키 전체에 종속되어야 함
        
    - 제 3정규형(3NF)
        
        → 2NF 만족하면서
        
        → 이행적 종속 제거
        
        → 일반 컬럼이 다른 일반 컬럼에 의존하면 안 됨
        
    - 반정규화
        - 조회 성능 향상을 위해 의도적으로 데이터 중복을 허용하는 방법
        - 특징
            
            → JOIN 감소
            
            → 조회 속도 향상
            
        - 단점
            
            → 갱신 이상 발생 가능
            
            → 데이터 일관성 관리 필요
            
- SQL
    - 데이터베이스와 소통하기 위한 언어, 선언형 언어
    - DDL
        - 데이터 구조를 정의하는 언어
        - CREATE → 테이블 생성
        - ALTER → 테이블 수정
        - DROP → 테이블 삭제
    - DML
        - 데이터를 조작하는 언어
        - INSERT → 데이터 삽입
        - SELECT → 데이터 조회
        - UPDATE → 데이터 수정
        - DELETE → 데이터 삭제
    - DCL
        - 데이터 접근 권한을 관리하는 언어
        - GRANT
            
            → 권한 부여
            
        - REVOKE
            
            → 권한 회수
            
    - SELECT* 사용 지양 이유
        - 성능 저하 → 필요 없는 데이터까지 조회
        - 유지보수 어려움 → 컬럼 추가 시 예상치 못한 데이터 반환
        - 가독성 저하 → 어떤 컬럼을 사용하는지 알기 어려움
        - 따라서 필요한 컬럼만 명시해서 조회하는 것이 좋음
    - JOIN
        - 정규화를 통해 분리된 여러 테이블을 다시 연결하여 조회하는 기능
        - 사용 이유
            
            → 여러 테이블의 데이터를 하나의 결과처럼 조회하기 위해
            
        - INNER JOIN : 양쪽 테이블에 모두 존재하는 데이터만 조회, 교집합 개념
        - LEFT JOIN : 왼쪽 테이블 데이터는 모두 조회, 연결되는 데이터가 없으면 NULL 반환
        - RIGHT JOIN : 오른쪽 테이블 데이터는 모두 조회, 연결되는 데이터가 없으면 NULL 반환
        - JOIN 주의사항
            
            → JOIN 컬럼에는 인덱스가 있는 것이 좋음
            
            → 너무 많은 JOIN은 성능 저하 발생 가능
            
    - VIEW
        - 실제 데이터를 저장하지 않는 가상의 테이블, SELECT 쿼리를 저장하여 테이블처럼 사용
        - 사용 이유
            
            → 복잡한 조회 로직 재사용
            
            → 유지보수 편리
            
            → 조회 로직 단순화
            
    - UNION
        - 여러 SELECT 결과를 하나로 합치는 기능
        - 특징 : 수직 결합, JOIN은 수평 결합
        - 사용 조건 : 컬럼 개수 동일, 컬럼 순서 동일, 데이터 타입 동일
- ERD
    - 데이터베이스 구조를 시각적으로 표현한 다이어그램
    - 사용 이유
        - 데이터 구조를 한눈에 파악하기 위해
        - 엔티티 간 관계를 명확하게 설계하기 위해
    - 구성 요소
        - 엔티티
            
            → 데이터를 저장하고 관리할 대상
            
            → 데이터베이스의 테이블에 해당
            
        - 속성
            
            → 엔티티가가지는 정보
            
            → 데이터의 컬럼에 해당
            
        - 관계
            
            → 엔티티 간 연관 관계를 표현
            
    - 관계 종류
        - 1:1 (One To One)
            
            → 하나의 데이터가 하나의 데이터와만 연결되는 관계
            
        - 1:N (One To Many)
            
            → 하나의 데이터가 여러 데이터와 연결되는 관계
            
        - 다:다(Many To Many)
            
            → 양쪽 모두 여러 데이터와 연결될 수 있는 관계
