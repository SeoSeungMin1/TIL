## 배운 내용

- 인터페이스
    - 클래스 간에 “이 기능을 반드시 제공해야 한다.”는 명시적 요구사항의 집합
    - 메서드의 이름과 매개변수만을 정의, 실제 동작 포함 x
    - 객체 생성 x → 구현이 완성된 객체 설계도가 아니라 계약이라서 단독으로 인스턴스 만들 수 없음.
    - 사용 이유 : 공통 기능 규칙을 정의해서 유연하고 확장성 있게 만들기 위함.
    - implements를 통해 구현.
    - 인터페이스 vs 추상 클래스
        
        → 인터페이스 : 규칙 중심, 동작 없음 / implements
        
        → 추상클래스 : 규칙 + 기본동작 작성 가능 / extends, 필요한 메서드만 abstract 붙이면 됨.
        
- 캡슐화
    - 객체의 내부 상태를 외부에서 직접 접근하지 못하도록 감추고, 메서드를 통해 간접적으로 접근하도록 하는 객체지향 개념
    - 사용 방법
        
        → 클래스 필드를 private으로 선언
        
        → getter(값 조회 메서드)와 setter(값 변경 메서드) 메서드 정의
        
    - getter, setter 메서드는 캡슐화를 구현하는 대표적인 관례, 모든 필드에 setter를 만드는 건 좋은 설계가 아니다.
    - 행위 중심 메서드 사용하는 것이 좋은 캡슐화.
- Generic
    - 데이터 타입을 클래스나 메서드가 사용할 때까지 지정하지 않고, 나중에 지정할 수 있도록 해주는 타입 매개변수 문법
    - 잘못된 타입 사용을 컴파일 시점에 막아주는 장치.
    - 제네릭 사용하면 오류가 런타임이 아니라 컴파일 타임에 드러난다.
    - 타입안정성, 잘못된 형변환이나 런타임 오류를 예방.
    - T는 기본타입인 int, double 직접 받을 수 없음
        
        → List<int>가 아니라 List<Integer>로 받아야함. wrapper 타입 사용
        
    - 사용 이유 : 하나의 코드를 다양한 타입에 재사용하면서도 컴파일 시점에 타입 오류를 잡기 위해서.
    - <T> : Type, <E> : Element, <K> : Key, <V> : Value
    - 타입 안정성이랑 코드 재사용성 동시에 확보하기 위함.
    - Object 쓸 수도 있는데 꺼낼 때 강제 형변환 해야함.→ 이 문제를 해결하려고 제네릭 사용.
    - 제네릭 불공변 → String이 Object의 하위 타입이지만 List<String>은 List<Object>의 하위 타입x
- 와일드 카드
    - 제네릭은 List<String>처럼 타입 고정.
    - 와일드 카드는 타입 범위를 조금 열어줘야함. → '?', '? extends', '? super'
    - 제네릭을 사용할 때 받을 수 있는 타입의 범위를 지정할 수 있도록 도와주는 문법
    - '?' : 불특정한 어떤 타입이든 허용하는 표현
    - '? super T' : T 또는 그 상위 타입만 허용
    - '? extends T' : T 또는 그 하위 타입만 허용
        
        → 제네릭에서 extends는 클래스 상속만이 아니라 인터페이스 구현까지 포함하는 넓은 의미.
        
    - List<Object>와 List<?>의 차이점
        
        → List<Object> : Object 타입요소를 받는 리스트
        
        → List<?> : 타입 인자를 알 수 없는 어떤 리스트(타입 안정성 때문에 임의의 요소 추가x)
        
    - List<?>, List<? extends T> ← 상한 제한이 있는 경우 읽기(get)가능, 추가(add) x
    - 사용 이유 : 다양한 타입을 유연하게 다룰 수 있도록 하기 위해.
- 컬렉션 인터페이스 - List
    - 컬렉션 인터페이스 : 자바에서 데이터의 집합을 다루기 위한 표준 구조를 정의한 상위 타입
    - 순서가 유지되며, 중복을 허용하는 요소의 집합을 표현하는 컬렉션 인터페이스
        
        → “순서가 유지된다” : 리스트에 순서대로 add()하면 그 순서를 보장한다.
        
        → “중복을 허용한다” : add(”A”)를 두 번 넣으면 두 개 모두 들어갈 수 있다. 중복적 요소 허용
        
        → List는 인터페이스. new List<>() 처럼 객체 직접 만들 수 없음.
        
        → ArrayList, LinkedList, Vector 같은 구현체 사용(대부분 ArrayList 사용)
        
        → ArrayList vs LinkedList
        
        | 비교 | ArrayList | LinkedList |
        | --- | --- | --- |
        | 구조 | 배열 | 연결 노드 |
        | 조회 | 빠름 | 느림 |
        | 중간 삽입/삭제 | 느림 | 빠름 |
    - 기본 문법 → List<String> names = new ArrayList<>();
    - 요소 삽입 → add(), 요소 조회→ get(), 요소 수정 → set(), 요소 삭제 → remove()
- 컬렉션 인터페이스 - Set
    - 중복을 허용하지 않고, 순서를 보장하지 않는 요소들의 집합
        
        → 중복 요소를 포함하지 않는 컬렉션. e1.equals(e2)가 참인 두 요소 쌍을 포함x
        
        → 중복을 허용하지 않는다. 같은 값을 두 번 추가해도 한 번만 저장된다.
        
        → 순서를 보장하지 않는다. Set은 무조건 순서가 없다는게 아니라 순서를 약속하지 않는다.
        
        → HashSet : 순서 보장 x , LinkedHashSet : 입력 순서 유지, TreeSet : 자동 정렬
        
    - 각 요소는 equals(), hashCode() 메서드 기준으로 유일성 판단 → 중복 여부 판단
    - equals() : 두 객체 내용이 같은지, hashCode() : 객체를 빠르게 찾기 위한 숫자값
    - 버킷 : HashSet이나 HashMap 내부에서 데이터를 저장하는 공간
        
        → hashCode() 값 기준으로 객체를 특정 버킷에 넣음
        
    - 사용 이유 : 중복 없는 데이터를 저장하고, 순서보다 유일성이 중요한 상황에서 간결하게 처리하기 위해
- 컬렉션 인터페이스 - Map
    - 하나의 키(key)에 하나의 값(value)을 대응시켜 저장하는 구조를 제공하는 컬렉션 인터페이스
    - List나 Set과 달리 Collection 인터페이스를 상속하지 않고 독립된 형태로 동작
    - 키와 값을 한 쌍으로 저장, 키는 중복 허용 하지 않고 값은 중복 허용.
    - 인덱스 기반이 아닌 키 기반의 접근 방식 사용 → 키로 쓰는 값은 불변에 가까운 값
    - HashMap : 순서 정보 기억x, LinkedHashMap : 순서 정보 기억, TreeMap : key 자동 정렬
    - 사용 이유 : 특정한 키를 통해 원하는 값을 빠르게 저장하고 조회할 수 있기 때문.
    - 요소 추가,수정 → put(K, V), 값 조회 → get(), 요소 삭제 → remove()

## 헷갈린 점

- 문법적인 내용을 배워서 기본 식이 계속 헷갈렸다.
- 컬렉션 인터페이스에서 Hash, Linked, Tree가 각각 어떤 차이가 있는지 헷갈렸다.

## 느낀점

각각의 문법을 정확하게 이해하고 있어야 헷갈리지 않고 코드 구현할 때 사용 할 수 있다고 생각하여 예제를 직접 타이핑 하고 실행 해보면서 이해 하는게 필요하다고 느꼈다.
