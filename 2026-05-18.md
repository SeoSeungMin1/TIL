## 배운 내용

- 인터페이스
    - 클래스 간에 “이 기능을 반드시 제공해야 한다.”는 명시적 요구사항의 집합
    - 메서드의 이름과 매개변수만 정의하며, 실제 동작은 포함하지 않는다.
    - 객체 생성 불가능 → 구현이 완성된 객체 설계도가 아니라 계약 역할이기 때문에 단독으로 인스턴스를 만들 수 없다.
    - 사용 이유 : 공통 기능 규칙을 정의하여 유연하고 확장성 있는 구조를 만들기 위해 사용한다.
    - `implements`를 통해 구현한다.
    - 인터페이스 vs 추상 클래스
        → 인터페이스 : 규칙 중심, 동작 없음 / `implements`
        → 추상 클래스 : 규칙 + 기본 동작 작성 가능 / `extends`, 필요한 메서드에만 `abstract` 사용
- 캡슐화
    - 객체의 내부 상태를 외부에서 직접 접근하지 못하도록 감추고, 메서드를 통해 간접적으로 접근하도록 하는 객체지향 개념
    - 사용 방법
        → 클래스 필드를 `private`으로 선언
        → getter(값 조회 메서드), setter(값 변경 메서드) 정의
    - getter, setter는 캡슐화를 구현하는 대표적인 관례지만, 모든 필드에 setter를 만드는 것은 좋은 설계가 아니다.
    - 행위 중심 메서드를 사용하는 것이 좋은 캡슐화 방식이다.
- Generic
    - 데이터 타입을 클래스나 메서드가 사용할 때까지 지정하지 않고, 나중에 지정할 수 있도록 하는 타입 매개변수 문법
    - 잘못된 타입 사용을 컴파일 시점에 막아주는 장치
    - 제네릭을 사용하면 오류가 런타임이 아니라 컴파일 타임에 드러난다.
    - 타입 안정성을 높이고, 잘못된 형변환이나 런타임 오류를 예방한다.
    - `T`는 기본 타입인 `int`, `double` 등을 직접 받을 수 없다.
        → `List<int>`가 아니라 `List<Integer>`처럼 wrapper 타입 사용
    - 사용 이유 : 하나의 코드를 다양한 타입에 재사용하면서도 컴파일 시점에 타입 오류를 잡기 위해 사용한다.
    - `<T>` : Type, `<E>` : Element, `<K>` : Key, `<V>` : Value
    - 타입 안정성과 코드 재사용성을 동시에 확보하기 위해 사용한다.
    - `Object`를 사용할 수도 있지만, 값을 꺼낼 때 강제 형변환이 필요하다. → 이 문제를 해결하기 위해 제네릭 사용
    - 제네릭 불공변
        → `String`은 `Object`의 하위 타입이지만 `List<String>`은 `List<Object>`의 하위 타입이 아니다.
- 와일드 카드
    - 제네릭은 `List<String>`처럼 타입이 고정된다.
    - 와일드 카드는 타입 범위를 조금 더 유연하게 열어주는 문법
        → `?`, `? extends`, `? super`
    - 제네릭을 사용할 때 받을 수 있는 타입의 범위를 지정할 수 있도록 도와준다.
    - `?`
        → 불특정한 어떤 타입이든 허용 
    - `? super T`
        → `T` 또는 그 상위 타입만 허용
    - `? extends T`
        → `T` 또는 그 하위 타입만 허용
        → 제네릭에서 `extends`는 클래스 상속뿐만 아니라 인터페이스 구현까지 포함하는 넓은 의미
    - `List<Object>`와 `List<?>`의 차이점
        → `List<Object>` : Object 타입 요소를 저장하는 리스트
        → `List<?>` : 타입 인자를 알 수 없는 리스트(타입 안정성 때문에 임의 요소 추가 불가능)
    - `List<?>`, `List<? extends T>`
        → 상한 제한이 있는 경우 읽기(`get`)는 가능하지만 추가(`add`)는 불가능
    - 사용 이유 : 다양한 타입을 유연하게 다루기 위해 사용한다
- 컬렉션 인터페이스 - List
    - 자바에서 데이터 집합을 다루기 위한 표준 구조를 정의한 컬렉션 인터페이스
    - 순서가 유지되며, 중복을 허용하는 요소의 집합
        → “순서가 유지된다” : `add()`한 순서를 유지
        → “중복을 허용한다” : 같은 값을 여러 번 저장 가능
        → `List`는 인터페이스이기 때문에 `new List<>()`처럼 직접 객체 생성 불가능
        → `ArrayList`, `LinkedList`, `Vector` 같은 구현체 사용 (대부분 `ArrayList` 사용)
        → `ArrayList` vs `LinkedList`
        | 비교 | ArrayList | LinkedList |
        | --- | --- | --- |
        | 구조 | 배열 | 연결 노드 |
        | 조회 | 빠름 | 느림 |
        | 중간 삽입/삭제 | 느림 | 빠름 |
    - 기본 문법
        → `List<String> names = new ArrayList<>();`
    - 요소 삽입 → `add()`
    - 요소 조회 → `get()`
    - 요소 수정 → `set()`
    - 요소 삭제 → `remove()`
- 컬렉션 인터페이스 - Set
    - 중복을 허용하지 않고, 순서를 보장하지 않는 요소들의 집합
        → 같은 값을 두 번 추가해도 한 번만 저장
        → Set은 무조건 순서가 없는 것이 아니라 “순서를 보장하지 않는다”는 의미
        → `HashSet` : 순서 보장 x
        → `LinkedHashSet` : 입력 순서 유지
        → `TreeSet` : 자동 정렬
    - 각 요소는 `equals()`, `hashCode()` 메서드를 기준으로 유일성을 판단
        → 중복 여부 판단
    - `equals()`
        → 두 객체의 내용이 같은지 비교
    - `hashCode()`
        → 객체를 빠르게 찾기 위한 숫자값
    - 버킷(bucket)
        → `HashSet`, `HashMap` 내부에서 데이터를 저장하는 공간
        → `hashCode()` 값을 기준으로 특정 버킷에 저장
    - 사용 이유 : 중복 없는 데이터를 저장하고, 순서보다 유일성이 중요한 상황을 간결하게 처리하기 위해 사용
- 컬렉션 인터페이스 - Map
    - 하나의 key에 하나의 value를 대응시켜 저장하는 컬렉션 인터페이스
    - `List`, `Set`과 달리 `Collection` 인터페이스를 상속하지 않고 독립적으로 동작
    - key와 value를 한 쌍으로 저장
        → key는 중복 허용 x
        → value는 중복 허용
    - 인덱스 기반이 아닌 key 기반 접근 방식 사용  
        → key는 불변에 가까운 값을 사용하는 것이 좋다.
    - `HashMap`
        → 순서 정보 기억 x
    - `LinkedHashMap`
        → 입력 순서 유지
    - `TreeMap`
        → key 자동 정렬
    - 사용 이유 : 특정 key를 통해 원하는 값을 빠르게 저장하고 조회할 수 있기 때문
    - 요소 추가/수정 → `put(K, V)`
    - 값 조회 → `get()`
    - 요소 삭제 → `remove()`
## 헷갈린 점
- 문법적인 내용을 많이 배우다 보니 기본 문법 구조가 계속 헷갈렸다.
- 컬렉션 인터페이스에서 `Hash`, `Linked`, `Tree` 계열의 차이점이 헷갈렸다.
## 느낀점
각 문법의 개념과 구조를 정확하게 이해하고 있어야 실제 코드 구현 과정에서 헷갈리지 않고 사용할 수 있다고 느꼈다. 단순히 읽는 것보다 예제를 직접 타이핑하고 실행 흐름을 확인하면서 학습하는 과정이 이해에 더 도움이 된다고 생각했다.
